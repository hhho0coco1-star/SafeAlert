# SafeAlert 프로젝트 기획서

> 공공데이터 기반 실시간 재난 알림 플랫폼

---

## 1. 프로젝트 개요

| 항목 | 내용 |
|------|------|
| 프로젝트명 | SafeAlert |
| 부제 | 공공데이터 기반 실시간 재난 알림 플랫폼 |
| 개발 목적 | 대규모 실시간 트래픽 처리, MSA 설계, 고가용성 시스템 구현 경험 |
| 기간 | 10~12주 |
| 인프라 | Kubernetes (minikube) |

---

## 2. 배경 및 목적

### 2.1 문제 정의

재난 발생 시 정보 전달 경로는 복잡하고 지연이 발생한다.

- 기상청, 행정안전부, 소방청 등 여러 기관에서 각각 데이터를 제공
- 사용자 입장에서 통합된 실시간 알림 채널 부재
- 재난 발생 시 수백만 명이 동시에 정보를 요청 → 대규모 트래픽 처리 필요

### 2.2 개발 목적

```
소방·안전 관리 현장 경험
        ↓
재난 정보 전달 지연의 위험성 인식
        ↓
기술로 문제 해결: 고가용성 실시간 알림 시스템 구현
```

### 2.3 기술적 학습 목표

| 번호 | 목표 |
|------|------|
| 1 | 고가용성의 확장 가능한 시스템 설계 및 운영 |
| 2 | 대규모 실시간 트래픽 처리 경험 |
| 3 | 장애 경험 및 복구 프로세스 수립 |
| 4 | Redis / Kafka 미들웨어 실무 활용 |
| 5 | 메시지 큐 기반 비동기 처리 구현 |
| 6 | 이벤트 기반 아키텍처 설계 |
| 7 | 분산 트랜잭션 / Saga / Outbox 패턴 적용 |
| 8 | timeout / retry / circuit breaker 설계 |
| 9 | 로그 · 메트릭 · 트레이싱 기반 관측 |
| 10 | Prometheus / Grafana 모니터링 구축 |

## 3. 사용자 및 이해관계자

### 3.1 사용자 유형

| 유형 | 설명 |
|------|------|
| 일반 사용자 | 재난 알림을 구독하고 실시간으로 수신하는 시민 |
| 관리자 | 알림 수동 발송, 시스템 모니터링, 통계 조회 |

### 3.2 사용자 스토리

**일반 사용자:**
- 나는 사용자로서 내 거주 지역의 재난 알림을 구독하고 싶다
- 나는 사용자로서 재난 발생 시 5초 이내에 알림을 받고 싶다
- 나는 사용자로서 알림 유형(기상, 지진, 화재 등)을 선택하여 구독하고 싶다
- 나는 사용자로서 과거 수신한 알림 이력을 조회하고 싶다

**관리자:**
- 나는 관리자로서 현재 알림 처리 현황을 실시간으로 모니터링하고 싶다
- 나는 관리자로서 특정 지역에 수동으로 알림을 발송하고 싶다
- 나는 관리자로서 시스템 장애 발생 시 즉시 알림을 받고 싶다

## 5. 비기능 요구사항

### 5.1 성능

| 항목 | 목표 |
|------|------|
| 알림 발송 지연 | 재난 감지 후 5초 이내 |
| 동시 접속 처리 | 최소 10,000 명 동시 WebSocket 연결 |
| API 응답 시간 | 95 percentile 기준 200ms 이하 |
| Kafka 처리량 | 초당 10,000 메시지 이상 |

### 5.2 가용성

| 항목 | 목표 |
|------|------|
| 시스템 가용성 | 99.9% (월 43분 이하 다운타임) |
| 핵심 서비스 Replica | Notification, Processor 최소 3개 |
| 자동 복구 | Pod 장애 시 30초 이내 재시작 |

### 5.3 확장성

- Kubernetes HPA로 CPU 70% 초과 시 자동 스케일 아웃
- 서비스별 독립 DB로 수평 확장 가능
- Kafka Consumer Group으로 처리량 선형 확장

### 5.4 관측 가능성

- 모든 서비스 요청에 TraceId 부여 (분산 트레이싱)
- 핵심 메트릭 수집: 처리량, 에러율, 지연시간, JVM 메모리
- 로그는 구조화된 JSON 형식으로 ELK 수집

## 6. 주요 시나리오 흐름

### 6.1 재난 알림 발송 흐름

```
1. Alert Collector가 공공 API를 주기적으로 폴링 (30초 간격)
2. 새로운 재난 정보 감지 시 Kafka 토픽(alert.raw)에 발행
3. Alert Processor가 메시지 소비
   - 지역 코드 매핑 (시/도/군/구 단위)
   - 중복 알림 제거 (Redis TTL 활용)
   - 카테고리 분류
4. 처리된 알림을 Kafka 토픽(alert.processed)에 발행
5. Notification Service가 소비
   - Subscription Service에서 해당 지역/카테고리 구독자 조회
   - 구독자 WebSocket 세션에 실시간 Push
6. 알림 발송 이벤트를 Event Store에 저장 (Outbox 패턴)
```

### 6.2 사용자 구독 등록 Saga 흐름

```
1. 사용자가 구독 등록 요청
2. Subscription Service가 DB에 구독 정보 저장 + Outbox 이벤트 발행
3. Kafka를 통해 Notification Service에 이벤트 전달
4. Notification Service가 WebSocket 세션 등록
5. 성공 확인 이벤트 발행 → Subscription Service에서 상태 업데이트
※ 실패 시 각 단계에서 보상 트랜잭션(Compensating Transaction) 실행
```

## 7. 제약 조건 및 가정

| 항목 | 내용 |
|------|------|
| 인프라 | 로컬 Kubernetes (minikube) — 실제 클라우드 비용 없음 |
| 공공 API | data.go.kr 무료 API 키 사용, 분당 호출 제한 있음 |
| 개발 인원 | 1인 개발 |
| 실제 트래픽 | 실제 수백만 사용자 없음 → k6 부하 테스트로 시뮬레이션 |
| 알림 채널 | SMS/이메일 제외, WebSocket + SSE만 구현 |
