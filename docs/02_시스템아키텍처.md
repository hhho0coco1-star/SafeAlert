---

> ⚠️ **설계안 vs 실제 구현**: 위 그림과 아래 2장의 일부는 초기 설계안이다.
> 실제 구현은 **7개 서비스**(API Gateway·Auth·Subscription·Notification·Alert Collector·Alert Processor·Frontend)이며,
> **Statistics·Admin은 독립 서비스 대신 Notification Service에 통합 구현**됐다.
> 포트·Replica의 실제 값은 README 서비스 구성 표와 각 `<service>/k8s/deployment.yaml`을 기준으로 한다.

## 2. 마이크로서비스 상세 정의

### 2.1 API Gateway

| 항목 | 내용 |
|------|------|
| 기술 | Spring Cloud Gateway |
| 역할 | 단일 진입점, 라우팅, JWT 인증 필터, Rate Limiting |
| Port | 8080 |
| Replica | 2 |

**주요 기능:**
- 요청을 각 마이크로서비스로 라우팅
- JWT 토큰 검증 (Auth Service 위임)
- IP 기반 Rate Limiting (Redis 활용)
- 요청/응답 로깅

### 2.2 Auth Service

| 항목 | 내용 |
|------|------|
| 기술 | Spring Boot 3.x, Spring Security, JWT |
| DB | PostgreSQL (auth_db) |
| Port | 8081 |
| Replica | 1 |

**주요 기능:**
- 회원가입, 로그인, 토큰 발급/갱신/취소
- Access Token (15분) + Refresh Token (7일) 전략
- Redis에 Refresh Token 저장

### 2.3 Alert Collector Service

| 항목 | 내용 |
|------|------|
| 기술 | Spring Boot 3.x, Spring Scheduler |
| 미들웨어 | Kafka Producer |
| Port | 8086 |
| Replica | 1 (단일 수집기) |

**주요 기능:**
- 공공 API 1시간 주기 폴링 (API quota 보호 — 초기 설계 30초에서 변경)
- 신규 재난 데이터 감지
- Kafka `alert.raw` 토픽에 이벤트 발행
- Circuit Breaker (Resilience4j): 공공 API 장애 시 차단
- 중복 수집 방지: Redis에 수집한 ID TTL 저장

**Circuit Breaker 설정:**
```
실패율 50% 이상 → OPEN (30초 대기)
OPEN → HALF_OPEN (3회 시도)
성공 시 → CLOSED
```

### 2.4 Alert Processor Service

| 항목 | 내용 |
|------|------|
| 기술 | Spring Boot 3.x |
| 미들웨어 | Kafka Consumer + Producer |
| DB | MongoDB (event_store) |
| Port | 8087 |
| Replica | 3 |

**주요 기능:**
- Kafka `alert.raw` 소비
- 지역 코드 파싱 및 매핑 (시/도/군/구)
- 알림 카테고리 분류
- 중복 알림 필터링 (Redis TTL 24시간)
- 처리된 알림을 `alert.processed` 토픽에 발행
- Event Store(MongoDB)에 처리 이력 저장

### 2.5 Notification Service

| 항목 | 내용 |
|------|------|
| 기술 | Spring Boot 3.x, Spring WebSocket |
| 미들웨어 | Kafka Consumer, Redis Pub/Sub |
| DB | PostgreSQL (notification_db) |
| Port | 8083 |
| Replica | 1 |

**주요 기능:**
- Kafka `alert.processed` 소비
- Subscription Service에서 대상 구독자 조회
- WebSocket STOMP 프로토콜로 실시간 Push
- 다중 인스턴스 간 세션 동기화: Redis Pub/Sub
- 알림 발송 이력 저장
- **(통합)** 관리자 통계·수동 발송·최근 알림 조회 API 제공 — 독립 Statistics·Admin Service 대신 통합

### 2.6 Subscription Service

| 항목 | 내용 |
|------|------|
| 기술 | Spring Boot 3.x |
| 미들웨어 | Kafka Producer (Outbox 발행) |
| DB | PostgreSQL (subscription_db) |
| Port | 8085 |
| Replica | 1 |

**주요 기능:**
- 사용자 구독 지역/카테고리 CRUD (시/군/구 단위, 최대 10개)
- 지역 코드 기반 구독자 목록 조회 (상향 계층 매칭)
- Transactional Outbox 패턴 적용 (구독 변경 이벤트 발행)

> ※ 구독 등록 Saga는 설계안이었으나 @Transactional + Outbox 조합으로 원자성이 보장돼 스킵됨 (WBS Phase 3-A 참조).

### 2.7 Statistics / Admin 기능 (Notification·Auth Service에 통합)

> 초기 설계에서는 Statistics Service(MongoDB stats_db, CQRS Read 모델)와 Admin Service를
> 독립 마이크로서비스로 분리하려 했으나, **1인 개발 규모와 운영 복잡도를 고려해 별도 서비스를 만들지 않고
> 기존 서비스에 통합**했다. (ADR-002·ADR-007 참조)

| 기능 | 통합 위치 |
|------|----------|
| 알림 발송 통계 (요약/최근 알림 집계) | Notification Service (`/api/admin/stats`, `/api/admin/alerts`) |
| 관리자 수동 알림 발송 | Notification Service (`/api/admin/alerts/manual`) |
| 회원 관리 (목록·권한·검색) | Auth Service (`/api/auth/admin/users`) |

별도 `stats_db`·CQRS Read 모델은 구현하지 않고, Notification Service가 이력 DB를 직접 집계한다.

## 3. 이벤트 / 메시지 흐름

### 3.1 Kafka 토픽 설계

| 토픽명 | 파티션 수 | 복제 수 | 발행자 | 소비자 |
|--------|----------|--------|--------|--------|
| `alert.raw` | 3 | 1 | Alert Collector | Alert Processor |
| `alert.processed` | 6 | 1 | Alert Processor | Notification Service |

> ※ `notification.sent`·`saga.subscription`·`saga.subscription.reply`는 초기 설계안으로 **미구현**이다.
> 실제 운영 토픽은 `alert.raw`·`alert.processed` 2개이며, Saga는 Outbox로 대체되어 스킵됐다.

### 3.2 메시지 형식 (예시)

**alert.raw 메시지:**
```json

  "alertId": "uuid",
  "source": "KMA",
  "type": "WEATHER_WARNING",
  "rawData": { ... },
  "collectedAt": "2025-01-01T12:00:00Z"
}
```

**alert.processed 메시지:**
```json
{
  "alertId": "uuid",
  "category": "WEATHER",
  "severity": "HIGH",
  "regions": ["11", "11010", "11010101"],
  "title": "서울 폭설 경보",
  "content": "...",
  "issuedAt": "2025-01-01T12:00:00Z"
}
```

## 4. 데이터 저장소 전략

### 4.1 서비스별 DB 분리 (Database per Service)

```
Auth Service        → PostgreSQL (auth_db)
Subscription Service → PostgreSQL (subscription_db)
Notification Service → PostgreSQL (notification_db)
Alert Processor     → MongoDB (event_store)
```

### 4.2 Redis 활용 전략

| 용도 | Key 패턴 | TTL |
|------|----------|-----|
| Refresh Token | `token:refresh:{userId}` | 7일 |
| API Rate Limit | `ratelimit:{ip}` | 1분 |
| 중복 알림 방지 | `alert:dedup:{alertId}` | 24시간 |
| 알림 캐시 | `alert:cache:{regionCode}` | 5분 |
| WebSocket 세션 | `ws:session:{userId}` | 연결 유지 동안 |

### 4.3 CQRS 적용

- **Write Model**: Transactional Outbox → Kafka → Event Store (MongoDB) — 구현됨
- **Read Model**: (미구현) 별도 Read 전용 서비스/컬렉션 없이 Notification Service가 이력 DB를 직접 집계

> ※ CQRS는 설계안이며 실제로는 단순화됐다. Write 경로(Outbox→Kafka→Event Store)는 구현됐으나
> 별도 Read Model(Statistics Service)은 만들지 않았다. (ADR-007 참조)

## 5. 장애 복원력 설계

### 5.1 Circuit Breaker (Resilience4j)

```
Alert Collector → 공공 API 호출 시 Circuit Breaker 적용
  - 실패율 50% 초과 → OPEN
  - OPEN 상태에서 fallback: 마지막 수집 데이터 캐시 반환
  - 30초 후 HALF_OPEN → 재시도
```

### 5.2 Retry 정책

```
Notification Service → WebSocket 발송 실패 시:
  - 최대 3회 재시도
  - 지수 백오프: 1초, 2초, 4초
  - 3회 모두 실패 → Dead Letter Queue (alert.dlq)
```

### 5.3 Timeout 설정

| 서비스 간 호출 | Timeout |
|--------------|---------|
| API Gateway → 내부 서비스 | 3초 |
| Alert Collector → 공공 API | 5초 |
| Subscription → DB 조회 | 2초 |

### 5.4 의도적 장애 주입 (Chaos Engineering)

```bash
# Notification Service Pod 강제 종료
kubectl delete pod notification-service-xxx

# 확인: 30초 이내 자동 복구 여부
kubectl get pods -w

# Kafka lag 확인
kafka-consumer-groups.sh --describe --group notification-group
```

## 6. Kubernetes 구성

### 6.1 리소스 구성

```yaml
# 실제 매니페스트(<service>/k8s/deployment.yaml) 기준값
services:
  api-gateway:       replicas: 2, cpu: 500m, memory: 512Mi
  auth-service:      replicas: 1, cpu: 300m, memory: 256Mi
  alert-collector:   replicas: 1, cpu: 500m, memory: 512Mi
  alert-processor:   replicas: 3, cpu: 500m, memory: 512Mi
  notification-svc:  replicas: 1, cpu: 400m, memory: 512Mi
  subscription-svc:  replicas: 1, cpu: 300m, memory: 256Mi
  frontend:          replicas: 1
# ※ 기획 목표(notification·processor 최소 3)는 로컬 자원 한계로 notification은 1로 축소.
#   부하 시 확장은 Phase 6 HPA로 대응.
```

### 6.2 HPA (Horizontal Pod Autoscaler)

> ✅ Phase 6에서 metrics-server 활성화 후 api-gateway·alert-processor 대상으로 **구현·검증 완료**.

| 서비스 | 최소 Replica | 최대 Replica | CPU 임계값 |
|--------|------------|------------|-----------|
| Alert Processor | 2 | 5 | 70% |
| API Gateway | 2 | 5 | 70% |

**실측 결과** (AWS EC2 minikube): api-gateway에 CPU 부하 발생 시 사용률이 **133%/70%** 까지 올라
Pod가 **2 → 4로 자동 확장(스케일아웃)** 되는 것을 확인했다. (상세 kubectl 출력은 README 참조)

### 6.3 Namespace 구성

```
safealert-app    : 애플리케이션 서비스
safealert-infra  : Kafka, Redis, PostgreSQL, MongoDB
safealert-monitor: Prometheus, Grafana, Jaeger, ELK
```

## 7. 관측 가능성 (Observability) 스택

### 7.1 로깅 (ELK Stack)

```
각 서비스 (JSON 로그)
    → Filebeat (Log Collector)
    → Logstash (파싱/변환)
    → Elasticsearch (저장)
    → Kibana (시각화)
```

**로그 필드 표준:**
- `traceId`, `spanId`: 분산 추적 연계
- `service`: 서비스명
- `level`: INFO/WARN/ERROR
- `message`: 로그 내용

### 7.2 메트릭 (Prometheus + Grafana)

**수집 메트릭:**
- HTTP 요청 수, 응답 시간, 에러율 (Micrometer)
- Kafka Consumer Lag
- JVM 메모리/GC
- Redis Hit/Miss Rate
- WebSocket 활성 연결 수

**Grafana 대시보드:**
- 서비스별 요청/에러율 대시보드
- Kafka 처리량 대시보드
- JVM 대시보드

### 7.3 분산 트레이싱 (Jaeger)

```
API Gateway → [TraceId 생성]
    → 각 서비스는 TraceId를 헤더로 전파
    → Kafka 메시지에도 TraceId 포함
    → Jaeger UI에서 전체 요청 흐름 시각화
```
