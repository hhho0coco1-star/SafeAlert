# 포트폴리오 프로젝트별 예상 면접 질문

> 포트폴리오의 3개 프로젝트(**HashTrip · LocalQuest · SafeAlert**)를 **포트폴리오 페이지 파트 순서대로** 분석해, 각 파트에서 면접관이 던질 만한 질문과 답변 포인트를 정리했다.
> 형식: **❓ 예상 질문 → 💡 답변 포인트 → ⚠️ 함정/주의**
> 12번(면접 가이드)·13번(CS 기초)과 함께 보면 좋다. 이 문서는 **"내 프로젝트 자체"에 대한 꼬리질문**에 집중한다.
>
> ⚠️ 표시(`[확인 필요]`)가 붙은 항목은 실제 코드/기억을 확인해 답을 확정해야 하는 부분이다. 모르면 "확인해보겠습니다"가 거짓말보다 낫다.

# 프로젝트 한눈 비교표 (이건 외워두면 좋다)

| 항목 | HashTrip | LocalQuest | SafeAlert |
|------|----------|------------|-----------|
| 기간/인원 | 2026.02~03 / 5명 | 2026.03~04 / 5명 | 개인 프로젝트 |
| 내 역할 | 핵심 알고리즘 + 발표 | **팀 리더** + 풀스택 | 전체(1인) |
| 프레임워크 | Spring **Legacy** 5.3 | Spring **Legacy** | Spring **Boot** 3.x |
| 영속성 | **MyBatis** | **MyBatis** | **Spring Data JPA** + MongoRepository |
| 인증 | 세션 + OAuth2 | **JWT + 세션 이원화** | **JWT(Access+Refresh)** + Redis |
| 프론트 | JSP | **React SPA + JSP 혼용** | React 18 (Vite) |
| DB | Oracle | Oracle | PostgreSQL + MongoDB |
| 아키텍처 | 모놀리식(레이어드) | 모놀리식(레이어드) | **MSA 7개 서비스** |
| 대표 기술 | 병렬 수집(ExecutorService) | QR 현장 인증 | Kafka·WebSocket·K8s |

> 이 표가 곧 **"프로젝트들이 서로 어떻게 다르고 내가 무엇을 점점 더 깊이 해봤는지"**의 스토리라인이다.
> HashTrip(레거시·팀 일부) → LocalQuest(레거시·리더·React 도입) → SafeAlert(Boot·MSA·1인 전체) 로 **난이도와 책임 범위가 상승**하는 흐름.

# 1. HashTrip — 여행 성향 분석 플랫폼

## 1-1. 파트: 프로젝트 개요 / 기술 선택

- ❓ **"왜 Spring Legacy(Framework)를 썼나요? Boot가 아니라?"**
  - 💡 교육 과정/팀 합의로 Legacy를 사용. XML 설정, `web.xml`, DispatcherServlet을 직접 구성해보며 **Boot가 자동으로 해주는 것의 내부를 이해**하게 됐다고 답하면 좋다. (SafeAlert에서 Boot로 넘어가며 차이를 체감했다고 연결)
- ❓ **"왜 JSP(서버 렌더링)를 선택했나요?"**
  - 💡 페이지 단위 이동이 중심인 서비스라 SPA가 과했고, **개발 속도**를 우선해 JSP 서버 렌더링으로 충분하다고 판단.
- ❓ **"MyBatis를 쓴 이유? JPA는요?"**
  - 💡 성향 분석의 **태그 집계·여행지 매칭 같은 복잡한 JOIN 쿼리를 SQL로 직접 제어**하려고 MyBatis 선택. (JPA는 객체 중심이라 복잡 통계 쿼리는 오히려 불편)
  - ⚠️ "JPA를 모르나?"로 이어질 수 있음 → SafeAlert에서 JPA를 썼다고 받아치면 균형이 맞는다.
- ❓ **"OAuth2를 3종(Google·Kakao·Naver)이나 붙인 이유?"**
  - 💡 여행 서비스라 **가입 장벽을 낮추는 것**이 중요. 소셜 로그인 통합으로 진입 마찰 최소화.

## 1-2. 파트: 담당 역할 — 성향 분석 알고리즘 (★ 가장 깊게 물어봄)

- ❓ **"성향 분석 알고리즘을 설명해보세요."**
  - 💡 1:1 선택지 설문에서 각 질문의 선택 태그에 **가중치를 부여 → `Map<String,Integer>`에 합산 → 최다 누적 태그를 최종 여행 스타일로 결정**.
- ❓ **"처음엔 어떻게 구현했고 왜 바꿨나요?"** (트러블슈팅과 직결, ★ 단골)
  - 💡 초기엔 **`Set`으로 태그 유무만 비교** → 같은 태그가 여러 질문에서 선택돼도 1회로만 카운트 → 실제 최다 선택 태그가 아닌 게 결정되는 버그. **`Set`→`Map` 합산 방식**으로 전면 수정해 해결.
  - ⚠️ 꼬리질문: **"동점이면 어떻게 처리?"** `[확인 필요]` — 동점 시 우선순위(질문 순서/태그 ID 등)를 어떻게 정했는지 실제 코드를 확인해 답하라. 모르면 약점이 된다.
- ❓ **"가중치 값은 어떻게 정했나요? 근거가 있나요?"**
  - 💡 `[확인 필요]` — 질문별 중요도에 따라 부여했는지, 균등했는지 확인. "데이터 기반이 아니라 설계 시 정한 휴리스틱"이라면 솔직히 말하고, 개선 방향(행동 이력 기반 학습)으로 마무리(회고에 이미 언급).
- ❓ **"이 알고리즘의 시간복잡도는?"**
  - 💡 질문 수 × 선택 태그 수에 비례하는 O(n) 수준. Map 조회/갱신은 O(1).

## 1-3. 파트: 담당 역할 — 공공 데이터 병렬 수집 (★ 성능 질문)

- ❓ **"왜 병렬 처리(ExecutorService)를 도입했나요?"**
  - 💡 여행지 상세를 **순차 호출**하니 데이터가 많을수록 응답 시간이 누적(블로킹). 한국관광공사 API를 **6~12 스레드로 병렬 호출**해 70~80% 단축.
- ❓ **"스레드 개수 6~12는 어떻게 정했나요?"**
  - 💡 `[확인 필요]` — CPU 코어 수/외부 API rate limit을 고려한 실험값인지 확인. "무작정 늘리면 API rate limit·컨텍스트 스위칭으로 오히려 느려진다"는 트레이드오프를 언급하면 좋다.
- ❓ **"배치 INSERT를 300건 단위로 한 이유?"**
  - 💡 건건 INSERT는 DB 왕복이 많아 느림 → 묶어서 한 번에. 너무 크면 메모리·트랜잭션 부담 → 300건이 적정선. `[확인 필요]`(정확한 근거 수치 확인)
- ❓ **"병렬 처리 중 예외나 동시성 문제는 없었나요?"** (★ 깊은 질문)
  - 💡 핵심 포인트: ① 공유 자료구조(리스트 등)에 여러 스레드가 쓰면 thread-safe 컬렉션/동기화 필요, ② 한 스레드 실패가 전체를 막지 않도록 예외 격리, ③ `ExecutorService`는 작업 후 **반드시 `shutdown()`**(안 하면 스레드 누수). 실제로 어떻게 처리했는지 `[확인 필요]`.
- ❓ **"70~80% 개선은 어떻게 측정했나요?"**
  - 💡 수집 전체 소요 시간을 before/after로 비교. `[확인 필요]`(측정 방법이 단순 체감인지 로그 기반인지 확인 — 체감이면 솔직히).

## 1-4. 파트: 도메인 설계(ERD)

- ❓ **"태그를 장소에 문자열로 저장하지 않고 N:M 중간 테이블로 뺀 이유?"**
  - 💡 한 태그가 여러 장소에 **재사용**되고, 자동 분류의 **가중치·신뢰도(tag_weight, tag_confidence)**를 함께 저장해야 해서 `Place_Tag_Map` 분리.
- ❓ **"정규화를 어디까지 했나요? 성능을 위해 반정규화한 곳은?"**
  - 💡 `[확인 필요]` — 통계/추천 조회가 느린 곳에서 반정규화나 인덱스를 썼는지 확인.

## 1-5. 파트: 트러블슈팅 / 회고

- ❓ **"그 버그(Set 중복)를 어떻게 발견했나요? 테스트는 있었나요?"**
  - 💡 특정 입력에서 결과가 직관과 다른 걸 보고 추적. 테스트 코드가 없었다면 "수동 검증의 한계를 느껴 이후 SafeAlert에서 단위 테스트를 도입했다"로 성장 연결.
- ❓ **"Spring Boot로 마이그레이션하면 뭐가 달라지나요?"** (회고에 언급)
  - 💡 XML/`web.xml` 설정이 자동 구성으로 사라지고, 내장 톰캣·starter 의존성·`application.yml`로 단순화. 실제로 SafeAlert에서 경험했다고 연결.

# 2. LocalQuest — 미션 기반 O2O 로컬 플랫폼

## 2-1. 파트: 프로젝트 개요 / 기술 선택 (★ 인증 이원화가 핵심)

- ❓ **"JWT와 세션을 왜 이원화했나요?"** (★ 매우 자주)
  - 💡 클라이언트 성격이 달라서. **사용자·비즈니스용 React SPA는 stateless한 JWT**, **관리자용 JSP는 서버 세션**. 인증 방식을 억지로 통일하지 않고 각 클라이언트에 맞는 방식을 택함.
  - ⚠️ 꼬리: **"한 서비스에 인증이 두 개면 복잡하지 않나?"** → 맞다. 진입점(REST API vs JSP 페이지)이 명확히 분리돼 있어 혼선은 없었지만, 관리 포인트가 둘이 되는 단점은 인정.
- ❓ **"React와 JSP를 왜 섞었나요?"**
  - 💡 상호작용 많은 사용자/비즈니스 화면은 **SPA(React)**, 빠른 개발이 중요한 관리자 CRUD는 **서버 렌더링(JSP)**. 적재적소.
  - ⚠️ 단점: 빌드/배포 파이프라인이 둘, 코드 스타일 분리. 일관성↓.

## 2-2. 파트: 담당 역할 — QR 현장 인증 (★ 가장 깊게 물어봄)

- ❓ **"QR 인증이 어떻게 동작하나요?"**
  - 💡 매장 승인 시 **UUID 기반 `qrAuthKey`**가 발급되어 `LQ_LOCATION_QR`에 저장. 사용자가 현장에서 QR 스캔(`jsQR`로 디코딩) → `authKey`를 서버로 전송 → 서버가 **JWT에서 userId 추출 + authKey 검증** → 진행 상태 업데이트.
- ❓ **"QR 부정 사용을 어떻게 막나요?"** (★ 핵심, 보안 질문)
  - 💡 검증 단계에서 여러 분기: ① **운영 중지 매장**(IsActive) 차단, ② **이전 위치 미완료** 시 차단(순서 강제), ③ **시간 초과** 차단, ④ **이미 완료** 차단(중복 방지). 현장 방문이 핵심이라 이 검증이 서비스 신뢰성의 중심.
  - ⚠️ 꼬리: **"QR 사진을 캡처해서 집에서 스캔하면?"** → 이게 약점일 수 있다. 정직하게 한계를 인정하고 보완책 제시: GPS 좌표 대조(Kakao 좌표 보유), QR에 시간 기반 토큰(회전), 영수증 인증 병행 등. `[확인 필요]`(실제 어디까지 구현했는지)
- ❓ **"같은 QR을 두 사람이 동시에 스캔하면(동시성)?"**
  - 💡 진행 상태 업데이트가 사용자별(`LQ_USER_QUEST_PROGRESS`)로 분리돼 충돌이 적음. 단, 같은 사용자가 빠르게 두 번 누르면 중복 처리 가능성 → **멱등/유니크 제약**으로 방어했는지 `[확인 필요]`.
- ❓ **"영수증 인증(Python OCR)은 어떻게 연동했나요?"**
  - 💡 영수증이 필요한 퀘스트는 receipt 레코드 생성 후 인증 대기 → 별도 **Python OCR 서버**가 영수증을 판독. `[확인 필요]`(연동 방식: REST 호출? 정확도/실패 처리?)

## 2-3. 파트: 담당 역할 — 관리자 14페이지 / 인터셉터

- ❓ **"관리자 권한은 어떻게 통제했나요?"**
  - 💡 `RoleBasedPageInterceptor`로 **세션 검증 + ADMIN 역할 확인**, `userId=1` 마스터 계정 보호.
- ❓ **"회고에 '인터셉터 범위 밖 인증 공백을 발견했다'고 했는데?"** (★ 회고를 읽은 면접관이 파고들 지점)
  - 💡 인터셉터가 특정 URL 패턴에만 걸려 있어 **매핑 누락 경로로 인증 없이 접근 가능했던 공백**을 발견하고 범위를 보정. 정확한 사례를 `[확인 필요]`로 준비해두면 강한 보안 스토리가 된다.

## 2-4. 파트: 트러블슈팅 — Blob URL 메모리 누수 (★ 프론트/브라우저 깊이)

- ❓ **"메모리 누수를 어떻게 발견하고 해결했나요?"**
  - 💡 QR 탭 반복 진입 시 메모리 증가 → 원인: `URL.createObjectURL()`로 만든 Blob URL이 **`revokeObjectURL()` 전까지 메모리에 유지**되는데 `useEffect` cleanup이 없었음. → `useRef`로 URL 추적 + **cleanup에서 `revokeObjectURL()`** 호출, 새 이미지 로드 전 이전 것 먼저 해제. **DevTools Memory 탭**으로 누수 소멸 확인.
- ❓ **"useEffect cleanup이 언제 실행되나요?"** (React 지식 확인)
  - 💡 컴포넌트 unmount 시 + 의존성 변경으로 effect 재실행되기 직전. 그래서 리소스 해제·구독 취소에 쓴다.
- ❓ **"메모리 누수가 왜 생기나(일반론)?"**
  - 💡 안 쓰는데 참조가 남아 GC가 회수 못 하는 것. JS에서 흔한 원인: 해제 안 한 타이머/이벤트 리스너/객체 URL, 클로저가 잡은 큰 객체 등.

## 2-5. 파트: 팀 리더 / 협업

- ❓ **"팀 리더로서 어려웠던 점과 해결?"**
  - 💡 **API 파라미터 명 불일치 버그**를 겪은 뒤 **팀 컨벤션 문서화 + API-first(인터페이스 계약 먼저 정의)**로 개선. 풀스택이라 프론트·백 양쪽 관점에서 계약의 중요성을 체감.
- ❓ **"일정/역할 분담은 어떻게?"** / **"갈등은 어떻게 조율?"**
  - 💡 `[확인 필요]` — 구체 사례 1개를 준비(STAR: 상황-과제-행동-결과 구조로).

# 3. SafeAlert — 실시간 재난 알림 (MSA) · 포트폴리오 관점 질문

> 기술 내부 질문은 12번(면접 가이드)·13번(CS)에 상세. 여기서는 **포트폴리오/프로젝트 선택 관점**의 질문에 집중.

## 3-1. 파트: 프로젝트 개요 / 동기

- ❓ **"왜 팀 프로젝트 다음에 1인 프로젝트를 했나요?"**
  - 💡 팀에서는 내가 맡은 영역(알고리즘·QR) 위주라 **시스템 전체를 처음부터 끝까지 설계·운영해보고 싶었다**. 그래서 MSA·인프라·모니터링까지 혼자 끝까지.
- ❓ **"1인인데 MSA는 과하지 않나요?"** (★ 단골, 12번에도 있음)
  - 💡 학습 목표가 분산 시스템 경험. 단, 규모 작은 Statistics·Admin은 통합하고 Saga는 Outbox로 대체하는 등 **필요한 만큼만 분리**. (12번 6부 답변 참고)

## 3-2. 파트: 기술 깊이 (포트폴리오에서 바로 보이는 것들)

- ❓ **"5초 이내 알림이라는데 어떻게 보장하나요?"**
  - 💡 이벤트 기반(Kafka) 비동기 + WebSocket Push. 단, "보장"은 강한 표현 → **평균/목표치**로 말하고 외부 API 수집 주기(1시간)와 혼동 않기. (수집 주기와 전달 지연은 다른 축)
  - ⚠️ 함정: "1시간 주기 수집인데 어떻게 5초?" → **수집(1시간)과 수집 후 사용자 전달(5초)은 다른 단계**임을 명확히 구분해 답할 것.
- ❓ **"WebSocket 다중 인스턴스 문제"** → 12번 5부 대표 스토리로 답.
- ❓ **"Kafka·K8s를 실제로 돌려봤나, 캡처뿐인가?"**
  - 💡 AWS EC2 minikube에 7개 서비스 배포 + HPA 2→4 스케일아웃 실측(README 캡처). 데모 영상 파트 B에서 시연 예정.

## 3-3. 파트: 약점/검증 → 12번 6부와 13번으로 커버

- 테스트, 실트래픽, 보안 트레이드오프 등은 12번 6부 답변을 그대로 활용.

# 4. 프로젝트 비교 / 종합 질문 (★ 가장 변별력 있는 파트)

> "세 프로젝트를 관통하는 질문"은 지원자의 성장과 이해 깊이를 본다. 비교표를 근거로 답하라.

- ❓ **"세 프로젝트 중 가장 어려웠던 건? 왜?"**
  - 💡 SafeAlert(분산 환경 디버깅·인프라). 또는 LocalQuest(리더+QR 보안). 하나 골라 **구체 사례 1개**와 함께.
- ❓ **"MyBatis와 JPA 둘 다 써봤는데 차이와 선택 기준은?"** (★ 비교표 직결)
  - 💡 MyBatis(HashTrip·LocalQuest)는 **SQL을 직접 제어** → 복잡 통계/튜닝에 유리, 보일러플레이트 많음. JPA(SafeAlert)는 **객체 중심·생산성↑·DB 독립성**, 복잡 쿼리는 QueryDSL/native 필요. "단순 CRUD·도메인 중심이면 JPA, 복잡 집계 쿼리 비중이 크면 MyBatis"가 내 기준.
- ❓ **"Legacy Spring과 Spring Boot 차이를 직접 체감한 게 있나요?"**
  - 💡 Legacy(HashTrip·LocalQuest)는 XML·`web.xml`·DispatcherServlet 수동 설정 → Boot(SafeAlert)는 자동 구성·내장 톰캣·starter로 설정이 사라짐. **"Boot가 뭘 자동으로 해주는지"를 Legacy를 해봐서 안다**는 게 강점.
- ❓ **"인증을 프로젝트마다 다르게 했는데 차이는?"** (★ 비교표 직결)
  - 💡 HashTrip=세션+OAuth2, LocalQuest=**JWT+세션 이원화**, SafeAlert=**JWT(Access 15분+Refresh 7일, Redis 저장)**. 모놀리식에선 세션도 충분하지만 **MSA에선 stateless JWT가 유리**(서버 간 세션 공유 불필요)함을 직접 경험. JWT의 강제 만료 불가 단점은 Refresh를 Redis에 둬서 보완.
- ❓ **"비동기/병렬을 여러 방식으로 써봤는데?"**
  - 💡 HashTrip은 **ExecutorService 병렬**(여러 API를 동시에 빨리), SafeAlert는 **Kafka 비동기 + WebFlux 논블로킹**(서비스 간 느슨한 결합·고동시성). "병렬 처리(CPU/IO 동시 실행)"와 "비동기 메시징(시간·결합 분리)"은 목적이 다름을 구분해 설명.
- ❓ **"공공 API를 여러 번 다뤘는데(관광공사·재난) 배운 점은?"**
  - 💡 외부 API는 **느리고 불안정**하다는 전제로 설계해야 함 → 병렬 수집(HashTrip), Circuit Breaker·중복 필터·스케줄링(SafeAlert). 특히 SafeAlert에서 **HTTP 200인데 응답코드로 오류를 주는** 공공 API 함정을 경험(CB가 못 잡던 버그).
- ❓ **"다음에 더 하고 싶은 것/부족한 점은?"**
  - 💡 테스트 코드 강화(TDD), CI/CD 자동화, 실트래픽 기반 튜닝. 회고들과 일관되게.

# 부록. 프로젝트 질문 대응 3원칙

1. **모르면 "확인해보겠습니다"** — 지어내면 꼬리질문에서 무너진다. 특히 `[확인 필요]` 항목은 미리 코드를 열어 답을 확정해둘 것.
2. **항상 트레이드오프와 함께** — "왜 그 기술?"엔 장점만 말하지 말고 **단점/대안**도 한 문장 붙인다.
3. **성장 서사로 연결** — 세 프로젝트는 "레거시·부분 → 리더·React → Boot·MSA·1인 전체"로 책임과 난이도가 오르는 스토리. 개별 답변을 이 흐름에 얹으면 설득력이 커진다.
