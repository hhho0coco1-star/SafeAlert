# SafeAlert 데모 영상 시나리오 & 내레이션 대본

> 범용 포트폴리오용(개발 직군 + 비개발 직군 모두). 총 길이 약 3~4분.
> 말투: ~합니다체(차분하고 신뢰감 있게). 괄호 `( )` 안은 **화면 동작 지시**이므로 읽지 않음. `[ ]`는 화면 자막 추천.

---

## 0. 영상 목표

| 목적 | 대상 | 증명 수단 |
|------|------|-----------|
| ① 기능이 실제로 동작한다 | 채용담당자/면접관 | 화면에서 알림이 진짜 뜨는 장면 |
| ② 어려운 기술을 제대로 썼다 | 기술 면접관 | MSA·Kafka·WebSocket이 만든 결과 |

**핵심 전략:** "실시간 알림이 날아오는 순간"을 클라이맥스로 잡고 거꾸로 설계한다.
가장 임팩트 있는 장면 = 관리자가 발송한 알림이 파이프라인을 거쳐 사용자 화면에 실시간 도착하는 분할 화면.

## 1. 접속 주소 한눈에 보기

촬영 중 주소창에 입력할 URL 모음. 프론트엔드는 `http://localhost:5173` 기준.

| 화면 | 주소(URL) | 접근 권한 |
|------|-----------|-----------|
| 랜딩 | `http://localhost:5173/` | 누구나 (로그인 상태면 대시보드로 리다이렉트) |
| 로그인/회원가입 | `http://localhost:5173/login` | 비로그인만 |
| 대시보드 | `http://localhost:5173/dashboard` | 로그인 필요 |
| 구독 설정 | `http://localhost:5173/subscriptions` | 로그인 필요 |
| 알림 이력 | `http://localhost:5173/history` | 로그인 필요 |
| 내 계정 | `http://localhost:5173/profile` | 로그인 필요 |
| 관리자 | `http://localhost:5173/admin` | ADMIN 계정만 |

**장면 5(기술) 보조 화면 주소:**

| 도구 | 주소(URL) | 비고 |
|------|-----------|------|
| Swagger UI (Auth) | `http://localhost:8081/swagger-ui/index.html` | API 명세 |
| Grafana | `http://localhost:3000` | admin/admin · K8s 포트포워딩 필요 |
| Prometheus | `http://localhost:9090` | K8s 포트포워딩 필요 |
| Jaeger | `http://localhost:16686` | 분산 트레이싱 · K8s 포트포워딩 필요 |

> 모니터링 도구는 K8s 포트포워딩이 필요하다(README "모니터링" 섹션 참고). 촬영 전 미리 띄워둘 것.

## 2. 장면 구성 요약

| 장면 | 이동 주소 | 보여주는 것 | 시간 |
|------|-----------|------------|------|
| 0 인트로 | `/` (랜딩) | 서비스 정체성·핵심 수치 | 15s |
| 1 인증 | `/login` | 이메일 인증·JWT | 25s |
| 2 구독 | `/subscriptions` | 지역·카테고리 선택 | 30s |
| 3 ⭐실시간 | `/admin` \| `/dashboard` 분할 | Kafka→WebSocket 실시간 | 40s |
| 4 이력 | `/history` | 기록·필터 | 20s |
| 5 기술 | Swagger·Grafana·Jaeger \| 트러블슈팅 그래픽 | 증거 몽타주 + Redis Pub/Sub·k6 사연 | 45s |
| 6 아웃트로 | 아키텍처 다이어그램 | 전체 구조·스택 | 10s |

> 장면 3은 **3-A(라이브)** 와 **3-B(설명형 폴백)** 두 버전을 준비한다.
> 촬영 시 실시간 수신 타이밍이 맞으면 3-A, 아니면 3-B 사용.

### 🎬 장면 3-A · 실시간 알림 [라이브 버전] (약 40초) ⭐
**📍 이동: 좌우 분할 — 왼쪽 창 `http://localhost:5173/admin`(관리자 계정), 오른쪽 창 `http://localhost:5173/dashboard`(일반 사용자 계정)**
**(화면: 좌우 분할. 왼쪽 관리자 `/admin`, 오른쪽 사용자 `/dashboard`)**

> 두 계정으로 동시에 로그인해야 하므로, 한쪽은 시크릿/다른 브라우저 창을 사용한다.
> ⚠️ **발송 지역 = 사용자 구독 지역을 반드시 일치시킬 것.** 대시보드는 구독 지역 채널만 듣기 때문에, 다른 지역에 발송하면 화면에 안 뜬다(정상 동작). 일치하는 지역(예: 서울)에 발송해야 알림이 뜬다.

> "이제 SafeAlert의 심장입니다.
> **(오른쪽 대시보드 상단 초록 뱃지 클로즈업)** 사용자 화면은 WebSocket으로 서버와 실시간 연결돼 있습니다. '실시간 연결됨' 표시가 보이시죠.
> **(왼쪽 관리자 화면. 발송 폼 입력)** 관리자가 서울 지역에 호우경보 알림을 작성하고, 발송합니다.
> **(발송 버튼 클릭 → 오른쪽 화면 주목)** 그러면 — **(알림 팝업 등장)** 보이시나요? 1초도 안 돼 사용자 화면에 알림이 도착했습니다.
> 이 알림은 Kafka 이벤트 파이프라인을 거쳐, **서울을 구독한 사용자에게만** 정확히 전달된 것입니다."

`[자막: 발송 → Kafka 파이프라인 → WebSocket 실시간 수신 (5초 이내)]`

### 🎬 장면 5 · 기술 아키텍처 (약 45초)

> **설계 원칙:** 기술을 *나열*하지 않고 *증명*한다. 영상은 "돌아가는 화면"을 보여주고, 기술 상세는 README·영상 설명란·면접이 받는다.
> 장면 5는 두 부분으로 나눈다 — 5-1(빠른 증거 몽타주) + 5-2(트러블슈팅 딥다이브 1건).

#### 장면 5-1 · 빠른 증거 몽타주 (약 20초)
**📍 이동: 순서대로 — `http://localhost:8081/swagger-ui/index.html` → `http://localhost:3000`(Grafana) → `http://localhost:16686`(Jaeger)**
**(화면: 각 화면을 빠르게 전환. 기술명이 아니라 '결과'를 말한다)**

> **(Swagger UI 스크롤)** "6개의 마이크로서비스가 각자 독립된 API로 동작하고,
> **(Grafana 대시보드 — 그래프가 움직이는 화면)** 실시간으로 메트릭이 수집되며,
> **(Jaeger — 요청 하나가 여러 서비스를 거치는 trace 화면)** 요청 하나가 서비스들을 거치는 경로까지 추적됩니다."

`[자막: 6개 MSA · Prometheus/Grafana 모니터링 · Jaeger 분산 트레이싱]`

#### 장면 5-2 · 트러블슈팅 딥다이브 (약 25초) ⭐
**📍 이동: 아키텍처 다이어그램(README) + Redis Pub/Sub 흐름 그래픽 + k6 결과 표(docs/06_부하테스트_결과.md 또는 캡처)**
**(화면: 말로 다 설명하지 말고 '문제 → 원인 → 해결'을 그래픽으로 보여준다)**

> **(아키텍처 그림에서 Notification 서버 3개 강조)** "알림 서버를 3대로 늘리자 문제가 생겼습니다. 사용자가 A서버에 연결됐는데 알림은 B서버가 처리하면, 알림이 전달되지 않았죠.
> **(Redis Pub/Sub 흐름 그래픽)** Redis Pub/Sub을 도입해, 어느 서버가 처리하든 모든 서버에 방송하도록 해결했습니다.
> **(k6 결과 표)** 그리고 k6로 동시 100명 부하 테스트를 돌려, 커넥션 풀을 튜닝해 응답 속도를 30퍼센트 개선했습니다."

`[자막: 다중 서버 알림 누락 → Redis Pub/Sub 해결 · k6 부하테스트로 30% 개선]`

> **이 사연을 고른 이유:** 장면 3(실시간 알림)과 직접 이어진다. "방금 본 실시간 알림을, 서버 여러 대에서도 안 끊기게 하려고 이렇게 해결했다"는 하나의 스토리가 된다.
> **영상에 다 담지 않는 기술(Outbox·Circuit Breaker·CQRS 등)** 은 영상 설명란/엔딩 슬라이드에 텍스트로, 또는 면접에서 설명한다. 상세 근거는 [docs/06_기술결정근거_ADR.md](06_기술결정근거_ADR.md) 참고.
> 비개발 직군 대상으로만 쓸 경우 장면 5 전체를 생략하거나 5-1만 사용 가능.

## 4. 촬영 전 체크리스트

- [ ] 전체 서버 기동 (`/run` 또는 README 로컬 실행 방법 참고)
- [ ] 장면 3 라이브 시연 가능 여부 사전 테스트 (관리자 발송 → 사용자 수신 타이밍 확인)

- [ ] ⚠️ **발송 지역 = 구독 지역 일치 필수** — 대시보드는 `/topic/alerts/{구독지역}`만 듣는다. 사용자가 서울만 구독했는데 관리자가 부산에 발송하면 화면에 안 뜬다(버그 아님, 지역 필터링 정상 동작). 모르고 찍으면 당황함
- [ ] (선택) **지역 필터링 역연출** — 먼저 타 지역(부산)에 발송 → 서울 사용자 화면 무반응 → "내 지역이 아니면 오지 않습니다" → 이어서 서울에 발송 → 즉시 팝업. 필터링이 실제 작동함을 눈으로 증명
- [ ] 장면 4 이력 API(`GET /api/notifications`) 실제 동작 여부 확인 — 방금 받은 알림이 이력 목록에 뜨는지 (안 뜨면 Mock Fallback 상태)
- [ ] 데모용 계정 2개 준비: 일반 사용자 1, 관리자(role=ADMIN) 1
- [ ] 이메일 인증 장면은 길면 배속 처리
- [ ] 화면 녹화 해상도/브라우저 줌 확인 (자막 가독성)
- [ ] **장면 5-1**: Swagger·Grafana·Jaeger 미리 띄워두기 (Grafana·Jaeger는 K8s 포트포워딩 필요)
- [ ] **장면 5-2**: Redis Pub/Sub 흐름 그래픽 1장 + k6 결과 표(docs/06) 캡처 준비
- [ ] 영상 설명란/엔딩 슬라이드용 기술스택 전체 목록 + ADR 링크 정리
