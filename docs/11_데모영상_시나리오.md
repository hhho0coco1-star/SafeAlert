# SafeAlert 데모 영상 시나리오 & 내레이션 대본

> 범용 포트폴리오용(개발 직군 + 비개발 직군 모두). 총 길이 약 4~5분.
> 말투: ~합니다체(차분하고 신뢰감 있게).
> **파트 A (로컬)** + **파트 B (AWS EC2 K8s)** 두 파트로 나눠 촬영 후 편집에서 합친다.

---

## 0. 영상 목표

| 목적 | 대상 | 증명 수단 |
|------|------|-----------|
| ① 기능이 실제로 동작한다 | 채용담당자/면접관 | 화면에서 알림이 진짜 뜨는 장면 |
| ② 어려운 기술을 제대로 썼다 | 기술 면접관 | MSA(마이크로서비스 아키텍처)·Kafka(카프카, 분산 메시지 스트리밍)·WebSocket(웹소켓, 실시간 양방향 통신)이 만든 결과 |

**핵심 전략:** "실시간 알림이 날아오는 순간"을 클라이맥스로 잡고 거꾸로 설계한다.
가장 임팩트 있는 장면 = 관리자가 발송한 알림이 파이프라인을 거쳐 사용자 화면에 실시간 도착하는 분할 화면.

## 2. 장면 구성 요약

### 🖥️ 파트 A — 로컬 촬영 (내 컴퓨터)
> 켜야 할 것: docker-compose + Spring Boot(스프링 부트) 6개 + npm run dev

| 장면 | 찍는 화면 | 보여주는 것 | 시간 |
|------|-----------|------------|------|
| 0 인트로 | 브라우저 `/` | 서비스 정체성·핵심 수치 | 15s |
| 1 인증 | 브라우저 `/login` | 이메일 인증·JWT(제이더블유티, 인증 토큰) | 25s |
| 2 구독 | 브라우저 `/subscriptions` | 지역·카테고리 선택 | 30s |
| 3 ⭐실시간 | 브라우저 `/admin` \| `/dashboard` 분할 | Kafka→WebSocket 실시간 알림 | 40s |
| 4 이력 | 브라우저 `/history` | 기록·필터 | 20s |
| 5 기술 | 브라우저 Swagger·Grafana·Jaeger | 증거 몽타주 + 트러블슈팅 | 45s |

> 장면 3은 **3-A(라이브)** 와 **3-B(설명형 폴백)** 두 버전을 준비한다.
> 촬영 시 실시간 수신 타이밍이 맞으면 3-A, 안 맞으면 3-B 사용.

### ☁️ 파트 B — AWS 촬영 (EC2 서버)
> 켜야 할 것: AWS EC2(이씨투, Amazon 가상 서버) 인스턴스 시작 → SSH(에스에스에이치, 원격 서버 접속) 접속
> 로컬 서버는 꺼도 됨

| 장면 | 찍는 화면 | 보여주는 것 | 시간 |
|------|-----------|------------|------|
| 6 ⭐Pod 전체 | EC2 SSH 터미널 | `kubectl get pods -A` — 23개 Pod(파드, 컨테이너 실행 단위) Running | 20s |
| 7 ⭐HPA 스케일아웃 | EC2 SSH 터미널 | CPU 부하 → REPLICAS(복제본 수) 2→4 자동 확장 | 40s |
| 8 아웃트로 | EC2 SSH 터미널 (마무리 멘트) | 전체 구조·스택 | 10s |

> 파트 A 촬영 완료 후 별도로 촬영. 편집에서 파트 A 뒤에 붙인다.

### 🎬 장면 3-A · 실시간 알림 — 라이브 버전 (약 40초) ⭐
> 실시간 수신이 자연스럽게 될 때 사용. 안 되면 3-B 사용.
1. 브라우저 창 2개 준비 (한쪽은 시크릿 모드)
   🎙️ "이제 SafeAlert의 심장입니다."
2. 왼쪽 창: `http://localhost:5173/admin` → 관리자 계정으로 로그인
3. 오른쪽 창: `http://localhost:5173/dashboard` → 일반 사용자 계정으로 로그인
4. 오른쪽 대시보드 상단 초록 뱃지(실시간 연결됨) 클로즈업
   🎙️ "사용자 화면은 WebSocket(웹소켓)으로 서버와 실시간 연결돼 있습니다. 실시간 연결됨 표시가 보이시죠."
5. 왼쪽 관리자 화면에서 발송 폼 입력 → 발송 버튼 클릭
   🎙️ "관리자가 서울 지역에 호우경보 알림을 작성하고, 발송합니다."
6. 오른쪽 화면으로 시선 이동 → 알림 팝업 등장 순간 촬영
   🎙️ "보이시나요? 1초도 안 돼 사용자 화면에 알림이 도착했습니다. 이 알림은 Kafka(카프카) 이벤트 파이프라인을 거쳐, 서울을 구독한 사용자에게만 정확히 전달된 것입니다."
> ⚠️ **발송 지역 = 사용자 구독 지역 반드시 일치.** 사용자가 서울 구독 → 관리자도 서울에 발송. 다른 지역에 발송하면 알림이 안 뜬다(버그 아님, 지역 필터링 정상 동작).

`[자막: 발송 → Kafka(카프카) 파이프라인 → WebSocket(웹소켓) 실시간 수신 (5초 이내)]`

## ☁️ 파트 B — AWS 촬영 대본
> **사전 준비:** AWS EC2(이씨투, Amazon Elastic Compute Cloud — Amazon 클라우드 가상 서버) 인스턴스 시작 → SSH(에스에스에이치, Secure Shell — 원격 서버에 안전하게 접속하는 방식) 접속 → `minikube status` 확인 → 터미널 폰트 크게 키우기
> **로컬 서버는 꺼도 됨.** 파트 A 촬영 완전히 끝낸 후 진행.

# 1단계: api-gateway Pod(파드) 이름 확인
kubectl get pods -n safealert-app | grep api-gateway
# 2단계: CPU(중앙처리장치) 부하 명령 (위에서 확인한 pod 이름으로 교체)

# yes: CPU를 계속 사용하게 만드는 명령 / > /dev/null: 출력 버림 / &: 백그라운드 실행
