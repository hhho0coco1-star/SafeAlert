### Step 1: minikube 클러스터를 왜 로컬에 구성하는가

#### 왜 이렇게 설계했나

실제 AWS EKS, GCP GKE는 시간당 비용이 발생한다. minikube는 로컬 PC에서 Kubernetes 클러스터를 그대로 시뮬레이션한다.

```
minikube (로컬)         vs       EKS (AWS)
─────────────────────────────────────────
비용: 없음                        시간당 과금
목적: 학습/개발                   프로덕션
K8s API: 동일                     동일
핵심 개념: 100% 동일              100% 동일
```

즉, minikube에서 익힌 `kubectl`, `Deployment`, `Service`, `HPA` 는 실제 클라우드에서도 그대로 동작한다. 학습 환경으로 minikube는 가장 현실적인 선택이다.

#### 파일 구조와 그 이유

```
infra/
└── (Phase 0에서는 파일 없음 — minikube 자체 설정만)
```

#### 검증 (직접 실행해보기)

```bash
# minikube 시작
minikube start --cpus=4 --memory=8192

# 클러스터 상태 확인
kubectl cluster-info

# 노드 확인 (minikube 노드 1개 Running 이어야 함)
kubectl get nodes

# 기대 출력값
# NAME       STATUS   ROLES           AGE   VERSION
# minikube   Ready    control-plane   1m    v1.xx.x
```

#### ✅ 이해 체크리스트

- [ ] minikube와 실제 K8s 클러스터(EKS)의 차이를 1줄로 설명할 수 있다
- [ ] `kubectl cluster-info` 출력에서 API Server URL을 찾을 수 있다
- [ ] 왜 이 프로젝트에서 Docker Compose 대신 Kubernetes를 선택했는지 말할 수 있다 (→ ADR-002 참고)

### Step 2: Namespace를 3개로 분리하는 이유

#### 왜 이렇게 설계했나

Namespace 없이 모든 Pod를 한 곳에 두면:
- 앱/인프라/모니터링 Pod가 뒤섞여 관리 불편
- `kubectl get pods` 명령이 수십 개 결과를 쏟아냄
- 나중에 팀 권한 분리가 불가능

```
safealert-app     → Auth, Gateway, Notification 등 애플리케이션 서비스
safealert-infra   → Kafka, Redis, PostgreSQL, MongoDB (미들웨어)
safealert-monitor → Prometheus, Grafana, Jaeger, ELK (모니터링)
```

각 Namespace를 독립적으로 관리하면 `kubectl get pods -n safealert-infra` 처럼 범위를 좁혀 볼 수 있다.

#### 파일 구조와 그 이유

```
infra/
└── namespaces.yaml     ← 3개 Namespace를 한 파일에 정의
```

```yaml
# namespaces.yaml 구조
apiVersion: v1
kind: Namespace
metadata:
  name: safealert-app
---
apiVersion: v1
kind: Namespace
metadata:
  name: safealert-infra
---
apiVersion: v1
kind: Namespace
metadata:
  name: safealert-monitor
```

#### 검증 (직접 실행해보기)

```bash
kubectl apply -f infra/namespaces.yaml

# 생성 확인
kubectl get namespaces

# 기대 출력 (safealert-* 3개 포함)
# NAME               STATUS   AGE
# safealert-app      Active   1s
# safealert-infra    Active   1s
# safealert-monitor  Active   1s
```

#### ✅ 이해 체크리스트

- [ ] Namespace 없이 모든 Pod를 default에 넣었을 때 생기는 문제를 설명할 수 있다
- [ ] `kubectl get pods -n safealert-infra` 와 `kubectl get pods` 의 차이를 안다
- [ ] 왜 앱/인프라/모니터링을 분리했는지 설명할 수 있다

### Step 3: Helm을 쓰는 이유

#### 왜 이렇게 설계했나

Kafka를 직접 `kubectl apply -f` 로 배포하려면 수십 개의 YAML 파일이 필요하다.

```
kubectl apply 방식
└── zookeeper-deployment.yaml
└── zookeeper-service.yaml
└── zookeeper-configmap.yaml
└── kafka-statefulset.yaml
└── kafka-service.yaml
└── kafka-configmap.yaml
... (계속)

Helm 방식
└── helm install kafka bitnami/kafka -f kafka/values.yaml
    → 위 모든 파일이 자동 생성됨
```

Helm은 Kubernetes용 패키지 매니저다. `values.yaml` 하나로 전체 설정을 관리한다.

#### 파일 구조와 그 이유

```
infra/
├── kafka/
│   └── values.yaml    ← Kafka Helm 차트의 설정값만 오버라이드
├── redis/
│   └── values.yaml
├── postgresql/
│   └── values.yaml
└── mongodb/
    └── values.yaml
```

기본 설정은 Helm 차트가 갖고 있고, `values.yaml`에는 이 프로젝트에 맞게 바꿀 값만 적는다.

#### 검증 (직접 실행해보기)

```bash
# Helm 저장소 추가
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update

# Kafka 배포
helm install kafka bitnami/kafka -n safealert-infra -f infra/kafka/values.yaml

# 배포 확인
kubectl get pods -n safealert-infra

# 기대 출력
# NAME                READY   STATUS    RESTARTS
# kafka-controller-0  1/1     Running   0
```

#### ✅ 이해 체크리스트

- [ ] Helm이 없을 때 Kafka 배포가 왜 복잡한지 설명할 수 있다
- [ ] `values.yaml`이 어떤 역할을 하는지 한 줄로 설명할 수 있다
- [ ] `helm install`, `helm upgrade`, `helm uninstall` 각각의 용도를 안다

### Step 4: Kafka 구조 이해

#### 왜 이렇게 설계했나

Alert Collector에서 Notification Service까지 메시지를 전달할 때 직접 HTTP 호출을 쓰면:

```
직접 HTTP 호출의 문제
├── Notification Service 다운 시 → 메시지 유실
├── Processor가 느리면 → Collector도 느려짐 (강결합)
└── 수평 확장 시 → 어느 인스턴스에 보낼지 복잡

Kafka 사용 시
├── Notification Service 다운 → 메시지는 Kafka에 보존
├── Processor가 느려도 → Collector는 계속 발행 (느슨한 결합)
└── Partition 수만큼 Consumer 병렬 처리
```

**핵심 개념 정리:**

| 개념 | 비유 | 역할 |
|------|------|------|
| Topic | 게시판 | 메시지 저장 카테고리 |
| Partition | 게시판 분류함 | 병렬 처리 단위 |
| Offset | 읽은 페이지 번호 | Consumer가 어디까지 읽었는지 |
| Consumer Group | 팀 | 같은 그룹은 파티션을 나눠서 처리 |

```
alert.processed 토픽 (파티션 6개)
├── Partition 0 → Notification Instance 1
├── Partition 1 → Notification Instance 2
├── Partition 2 → Notification Instance 3
├── Partition 3 → Notification Instance 1 (재할당)
├── Partition 4 → Notification Instance 2
└── Partition 5 → Notification Instance 3
```

#### 검증 (직접 실행해보기)

```bash
# Kafka Pod 접속
kubectl exec -it kafka-controller-0 -n safealert-infra -- bash

# 토픽 생성
kafka-topics.sh --create \
  --bootstrap-server localhost:9092 \
  --topic alert.raw \
  --partitions 3 \
  --replication-factor 1

# 토픽 목록 확인
kafka-topics.sh --list --bootstrap-server localhost:9092

# 메시지 발행 테스트
kafka-console-producer.sh --bootstrap-server localhost:9092 --topic alert.raw
> {"test": "hello kafka"}

# 다른 터미널에서 메시지 소비
kafka-console-consumer.sh --bootstrap-server localhost:9092 \
  --topic alert.raw --from-beginning
```

#### ✅ 이해 체크리스트

- [ ] 토픽, 파티션, Consumer Group의 관계를 그림으로 설명할 수 있다
- [ ] Notification Service를 3개 인스턴스로 늘렸을 때 파티션이 어떻게 배분되는지 설명할 수 있다
- [ ] Consumer Group이 없으면 어떤 일이 생기는지(팬아웃 vs 경쟁 소비) 설명할 수 있다
- [ ] Kafka와 RabbitMQ의 차이를 1줄로 설명할 수 있다 (→ ADR-001 참고)

### Step 5: Redis를 3가지 용도로 쓰는 이유

#### 왜 이렇게 설계했나

이 프로젝트에서 Redis는 한 가지가 아니라 3가지 역할을 한다.

```
역할 1: 캐시
  → 공공 API에서 수집한 최신 알림을 5분간 캐시
  → 같은 데이터를 DB에서 반복 조회하는 비용 제거

역할 2: 세션/토큰 저장
  → Refresh Token을 Redis에 저장 (TTL 7일 자동 만료)
  → DB에 저장하면 만료 토큰 삭제 배치가 필요 → Redis TTL로 대체

역할 3: Pub/Sub (WebSocket 세션 동기화)
  → Notification Service 인스턴스가 3개일 때 사용자 A가 인스턴스 1에 연결
  → 인스턴스 2에서 알림을 보내려면 인스턴스 1로 전달 필요
  → Redis Pub/Sub로 모든 인스턴스에 브로드캐스트
```

#### 검증 (직접 실행해보기)

```bash
# Redis CLI 접속
kubectl exec -it redis-master-0 -n safealert-infra -- redis-cli

# 캐시 저장/조회
SET alert:cache:11010 '{"title":"서울 경보"}' EX 300
GET alert:cache:11010
TTL alert:cache:11010

# Pub/Sub 테스트 (터미널 2개)
# 터미널 1: 구독
SUBSCRIBE ws:broadcast

# 터미널 2: 발행
PUBLISH ws:broadcast '{"userId":"abc","message":"재난 발생"}'
# 터미널 1에서 메시지 수신 확인
```

#### ✅ 이해 체크리스트

- [ ] Redis TTL이 Refresh Token 만료 처리에서 왜 DB보다 유리한지 설명할 수 있다
- [ ] Redis Pub/Sub가 Kafka와 다른 점(영속성 없음)과 그래도 여기서 쓰는 이유를 설명할 수 있다
- [ ] `EX 300`이 의미하는 바와 TTL이 0이 됐을 때 어떻게 되는지 안다

### Step 6: API Gateway가 왜 필요한가

#### 왜 이렇게 설계했나

API Gateway 없이 클라이언트가 직접 각 서비스를 호출하면:

```
클라이언트가 직접 호출하는 경우 (문제)
├── 클라이언트가 Auth 서비스 주소를 알아야 함
├── 클라이언트가 Subscription 서비스 주소를 알아야 함
├── JWT 검증을 각 서비스마다 구현해야 함
├── Rate Limiting을 각 서비스마다 구현해야 함
└── 서비스 주소가 바뀌면 클라이언트도 수정 필요

API Gateway를 통한 경우 (해결)
클라이언트 → API Gateway (8080)
                ├── /api/auth/** → Auth Service
                ├── /api/subscriptions/** → Subscription Service
                └── /api/notifications/** → Notification Service
                    ↑ JWT 검증, Rate Limit 한 곳에서 처리
```

#### 파일 구조와 그 이유

```
api-gateway/
├── src/main/resources/
│   └── application.yml         ← 라우팅 규칙 정의
└── src/main/java/.../
    └── filter/
        ├── JwtAuthFilter.java  ← JWT 검증 필터 (모든 요청에 적용)
        └── RateLimitFilter.java ← Redis 기반 요청 횟수 제한
```

#### 검증 (직접 실행해보기)

```bash
# Gateway를 통해 Auth Service 호출
curl -X POST http://localhost:8080/api/auth/signup \
  -H "Content-Type: application/json" \
  -d '{"email":"test@test.com","password":"test1234!","nickname":"테스터"}'

# JWT 없이 보호된 API 호출 → 401 응답 확인
curl http://localhost:8080/api/subscriptions
# 기대: {"success":false,"code":"AUTH_INVALID_TOKEN",...}
```

#### ✅ 이해 체크리스트

- [ ] API Gateway 없이 MSA를 운영할 때 생기는 3가지 이상의 문제를 말할 수 있다
- [ ] Spring Cloud Gateway의 필터가 언제 실행되는지 흐름을 설명할 수 있다
- [ ] Rate Limiting을 Redis로 구현하는 방법의 원리(슬라이딩 윈도우)를 설명할 수 있다

### Step 7: JWT Access/Refresh Token을 왜 분리하는가

#### 왜 이렇게 설계했나

```
Access Token만 사용하는 경우 (문제)
├── 유효기간 짧으면 → 자주 로그인 해야 함 (UX 나쁨)
└── 유효기간 길면 → 토큰 탈취 시 오랫동안 악용 가능 (보안 나쁨)

Access + Refresh Token 분리 (해결)
├── Access Token: 15분 (짧게, 탈취돼도 피해 최소화)
└── Refresh Token: 7일 (길게, Redis에 저장, 재발급에만 사용)

흐름:
1. 로그인 → Access(15분) + Refresh(7일) 발급
2. API 요청 → Access Token 사용
3. Access 만료 → Refresh Token으로 새 Access Token 발급
4. 로그아웃 → Redis에서 Refresh Token 삭제 (즉시 무효화)
```

#### 파일 구조와 그 이유

```
auth-service/src/main/java/.../
├── service/
│   ├── AuthService.java         ← 회원가입/로그인 비즈니스 로직
│   └── TokenService.java        ← 토큰 발급/검증/갱신 전담
├── security/
│   └── JwtProvider.java         ← JWT 생성/파싱 유틸리티
└── repository/
    └── RefreshTokenRepository.java  ← Redis에 Refresh Token 저장
```

#### 검증 (직접 실행해보기)

```bash
# 로그인으로 토큰 발급
curl -X POST http://localhost:8080/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"test@test.com","password":"test1234!"}'

# 응답에서 accessToken 복사 후 보호된 API 호출
curl http://localhost:8080/api/subscriptions \
  -H "Authorization: Bearer {복사한_accessToken}"

# Redis에서 Refresh Token 저장 확인
kubectl exec -it redis-master-0 -n safealert-infra -- redis-cli
KEYS token:refresh:*
```

#### ✅ 이해 체크리스트

- [ ] Access Token을 짧게 설정하는 보안적 이유를 설명할 수 있다
- [ ] Refresh Token을 DB 대신 Redis에 저장하는 이유를 말할 수 있다
- [ ] 로그아웃 시 Redis에서 Refresh Token을 삭제하는 게 왜 즉각적인 무효화가 되는지 설명할 수 있다

### Step 8: Database per Service — 왜 DB를 서비스별로 분리하는가

#### 왜 이렇게 설계했나

```
DB를 하나로 공유하는 경우 (문제)
├── Auth 서비스와 Subscription 서비스가 같은 DB 사용
├── Auth 서비스 배포 시 스키마 변경 → Subscription에도 영향
├── 어느 서비스가 DB 부하를 일으키는지 구분 불가
└── 마이크로서비스의 독립 배포 장점이 사라짐

Database per Service 패턴 (해결)
├── Auth DB: 사용자 정보만
├── Subscription DB: 구독 정보만
├── Notification DB: 발송 이력만
→ 각 서비스가 독립적으로 스키마 변경 가능
→ 서비스별 DB 성능 튜닝 가능
```

**트레이드오프:** 서비스 간 JOIN이 불가능해진다. 대신 API 호출 또는 이벤트로 데이터를 연동한다.

#### 파일 구조와 그 이유

```
infra/postgresql/
├── values.yaml              ← PostgreSQL Helm 설정
└── init-scripts/
    ├── 01-create-auth-db.sql        ← auth_db 스키마 초기화
    ├── 02-create-subscription-db.sql
    └── 03-create-notification-db.sql
```

#### 검증 (직접 실행해보기)

```bash
# PostgreSQL 접속
kubectl exec -it postgresql-0 -n safealert-infra -- psql -U postgres

# DB 목록 확인
\l
# 기대: auth_db, subscription_db, notification_db 모두 존재

# 각 DB 접속 후 테이블 확인
\c auth_db
\dt
```

#### ✅ 이해 체크리스트

- [ ] 하나의 DB를 공유할 때 생기는 문제를 MSA 관점에서 3가지 설명할 수 있다
- [ ] 서비스 간 JOIN이 필요한 상황에서 어떻게 해결하는지 (API 호출 또는 이벤트 기반) 설명할 수 있다
- [ ] 이 패턴의 단점(트랜잭션 복잡도 증가)도 함께 설명할 수 있다

### Step 9: Transactional Outbox 패턴

#### 왜 이렇게 설계했나

구독 등록 시 DB 저장과 Kafka 발행을 동시에 해야 한다. 이 둘은 하나의 트랜잭션으로 묶을 수 없다.

```
문제 시나리오 1: DB 저장 후 Kafka 발행 실패
→ DB에는 구독이 등록됐지만 Notification에 전달 안 됨 → 알림 미수신

문제 시나리오 2: Kafka 발행 후 DB 저장 실패
→ 알림은 전송됐지만 구독 정보가 없음 → 데이터 불일치

Transactional Outbox 해결책:
1. DB 트랜잭션 안에서 구독 저장 + outbox_events 테이블에 이벤트 함께 저장
2. 별도 스케줄러가 outbox_events의 PENDING 이벤트를 Kafka에 발행
3. 발행 성공 시 상태를 PUBLISHED로 업데이트

→ DB 저장과 Kafka 발행이 결과적으로 일치 (Eventual Consistency)
```

#### 파일 구조와 그 이유

```
subscription-service/src/main/java/.../
├── domain/
│   ├── Subscription.java       ← 구독 엔티티
│   └── OutboxEvent.java        ← Outbox 이벤트 엔티티
├── service/
│   └── SubscriptionService.java ← DB 저장 + Outbox 이벤트 동시 저장
└── scheduler/
    └── OutboxPublisher.java     ← 주기적으로 PENDING 이벤트를 Kafka 발행
```

#### 검증 (직접 실행해보기)

```bash
# 구독 등록 API 호출
curl -X POST http://localhost:8080/api/subscriptions/regions \
  -H "Authorization: Bearer {token}" \
  -H "Content-Type: application/json" \
  -d '{"regionCode":"11010"}'

# outbox_events 테이블에서 이벤트 확인
kubectl exec -it postgresql-0 -n safealert-infra -- psql -U postgres subscription_db
SELECT event_id, event_type, status, created_at FROM outbox_events ORDER BY created_at DESC LIMIT 5;

# 잠시 후 PUBLISHED로 상태 변경 확인
SELECT status FROM outbox_events WHERE event_type = 'SUBSCRIPTION_CREATED';
```

#### ✅ 이해 체크리스트

- [ ] Outbox 패턴 없이 DB 저장 + Kafka 발행을 할 때 생기는 2가지 장애 시나리오를 설명할 수 있다
- [ ] Outbox 테이블의 `status` 컬럼이 왜 필요한지 설명할 수 있다
- [ ] 이 패턴이 "Exactly Once"가 아닌 "At Least Once"인 이유와 중복 처리 방법을 설명할 수 있다

### Step 10: Alert Collector의 Circuit Breaker가 왜 필요한가

#### 왜 이렇게 설계했나

공공 API(기상청, 행정안전부)는 외부 서비스다. 언제든 느려지거나 다운될 수 있다.

```
Circuit Breaker 없는 경우 (문제)
├── 기상청 API 응답 지연 (30초 이상)
├── Alert Collector의 모든 스레드가 응답 대기
├── 다음 폴링도 지연 → 쌓이다가 OOM(메모리 부족) 발생
└── Alert Collector 서비스 전체 다운

Circuit Breaker 있는 경우 (해결)
├── 기상청 API 실패율 50% 초과 감지
├── Circuit OPEN → 기상청 API 호출 즉시 차단
├── Fallback: 마지막 수집 데이터 캐시 반환 (서비스 계속 동작)
└── 30초 후 HALF_OPEN → 소량 재시도 → 성공 시 CLOSED 복귀
```

**Circuit Breaker 상태 전환:**

```
CLOSED (정상) ──[실패율 ≥ 50%]──→ OPEN (차단)
                                        │
                                   [30초 대기]
                                        ↓
                              HALF_OPEN (제한 재시도)
                              ├──[성공]──→ CLOSED
                              └──[실패]──→ OPEN
```

#### 파일 구조와 그 이유

```
alert-collector/src/main/java/.../
├── client/
│   ├── WeatherApiClient.java       ← 기상청 API 호출 + Circuit Breaker 적용
│   └── DisasterApiClient.java      ← 행정안전부 API 호출
└── config/
    └── Resilience4jConfig.java     ← Circuit Breaker 임계값 설정
```

#### 검증 (직접 실행해보기)

```bash
# Actuator를 통해 Circuit Breaker 상태 확인
curl http://localhost:8080/api/actuator/circuitbreakers

# Grafana에서 circuit_breaker_state 메트릭 확인
# 또는 로그에서 "CircuitBreaker 'weatherApi' is OPEN" 메시지 확인
kubectl logs -n safealert-app deployment/alert-collector | grep CircuitBreaker
```

#### ✅ 이해 체크리스트

- [ ] Circuit Breaker가 없을 때 외부 API 장애가 어떻게 전체 시스템으로 전파되는지 설명할 수 있다
- [ ] CLOSED → OPEN → HALF_OPEN → CLOSED 전환 조건을 말할 수 있다
- [ ] Fallback 처리의 목적(서비스 완전 중단 방지)을 설명할 수 있다

### Step 11: Kafka 토픽을 3단계로 분리한 이유

#### 왜 이렇게 설계했나

```
1단계로 처리하는 경우 (문제)
Alert Collector가 바로 Notification Service에 발행
→ Collector가 지역 파싱, 중복 체크, 분류까지 모두 해야 함
→ 단일 책임 원칙(SRP) 위반
→ Collector 로직이 복잡해져 유지보수 어려움

3단계로 분리 (해결)
alert.raw       : 공공 API에서 받은 원시 데이터 그대로
alert.processed : 지역 매핑/분류/중복 제거 완료된 데이터
notification.*  : 실제 발송 대상 사용자 정보 포함

각 서비스가 한 가지 일만 담당:
Alert Collector   → alert.raw 발행 (수집만)
Alert Processor   → alert.raw 소비 → alert.processed 발행 (처리만)
Notification Svc  → alert.processed 소비 → 발송 (전달만)
```

#### 검증 (직접 실행해보기)

```bash
# Kafka UI (Kafka 클러스터에 포함된 경우) 또는 직접 확인
kubectl exec -it kafka-controller-0 -n safealert-infra -- \
  kafka-topics.sh --list --bootstrap-server localhost:9092

# 각 토픽의 파티션 수 확인
kafka-topics.sh --describe --bootstrap-server localhost:9092 --topic alert.raw
kafka-topics.sh --describe --bootstrap-server localhost:9092 --topic alert.processed

# Consumer Lag 확인 (처리 지연 여부)
kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
  --describe --group alert-processor-group
```

#### ✅ 이해 체크리스트

- [ ] 단일 토픽으로 처리할 때 생기는 문제(높은 결합도)를 설명할 수 있다
- [ ] Consumer Lag이 무엇이고 왜 모니터링해야 하는지 설명할 수 있다
- [ ] alert.processed의 파티션 수를 alert.raw보다 많이 설정한 이유를 설명할 수 있다

### Step 12: WebSocket 다중 인스턴스에서 Redis Pub/Sub를 쓰는 이유

#### 왜 이렇게 설계했나

```
문제 시나리오:
사용자 A가 Notification Instance 1에 WebSocket 연결
→ Kafka에서 알림 소비는 Instance 2가 담당
→ Instance 2는 사용자 A의 WebSocket 세션이 없음
→ 알림 전송 불가

Redis Pub/Sub 해결:
Instance 2가 알림 소비
→ Redis 채널에 알림 발행 (모든 인스턴스에 전파)
→ Instance 1이 Redis 채널 구독 중 → 메시지 수신
→ Instance 1이 사용자 A의 WebSocket 세션에 전송
```

```
흐름도:
Kafka → Instance 2 → Redis Pub/Sub 채널 "ws:broadcast"
                              ↓ (브로드캐스트)
                    Instance 1 (수신) → 사용자 A WebSocket
                    Instance 2 (수신) → 사용자 B WebSocket
                    Instance 3 (수신) → 사용자 C WebSocket
```

#### 파일 구조와 그 이유

```
notification-service/src/main/java/.../
├── websocket/
│   ├── WebSocketConfig.java        ← STOMP 엔드포인트 설정
│   └── NotificationHandler.java    ← WebSocket 메시지 핸들러
└── service/
    ├── NotificationService.java    ← Kafka 소비 + Redis 발행
    └── RedisSubscriber.java        ← Redis 채널 구독 → WebSocket 전송
```

#### 검증 (직접 실행해보기)

```bash
# 브라우저 개발자 도구 Console에서
const ws = new WebSocket('ws://localhost:8080/api/notifications/ws');
ws.onmessage = (e) => console.log('알림 수신:', e.data);

# 관리자 API로 수동 알림 발송 후 브라우저 콘솔에서 수신 확인
curl -X POST http://localhost:8080/api/admin/alerts/manual \
  -H "Authorization: Bearer {adminToken}" \
  -d '{"title":"테스트","content":"테스트 알림","targetRegions":["11"]}'
```

#### ✅ 이해 체크리스트

- [ ] 다중 인스턴스 환경에서 WebSocket 세션 불일치 문제가 왜 생기는지 설명할 수 있다
- [ ] Redis Pub/Sub와 Kafka의 차이(영속성 유무)와 여기서 Pub/Sub를 선택한 이유를 설명할 수 있다
- [ ] `ws:server:{userId}` Redis Key가 어떤 역할을 하는지 설명할 수 있다

### Step 13: Saga Choreography vs Orchestration

#### 왜 이렇게 설계했나

```
Orchestration (중앙 조율)
Saga Orchestrator가 각 서비스에게 명령
└── 장점: 흐름을 한 곳에서 파악 가능
└── 단점: Orchestrator가 단일 장애 지점(SPOF)이 됨

Choreography (이벤트 기반 분산 조율)
각 서비스가 이벤트를 듣고 자율적으로 반응
└── 장점: 중앙 의존 없음, 서비스 독립성 유지
└── 단점: 전체 흐름 파악이 어려움

이 프로젝트 선택: Choreography
이유: 서비스가 8개로 비교적 단순하고
     MSA의 독립성을 최대한 유지하는 게 학습 목적에 맞음
```

**구독 등록 Saga 흐름 (Choreography):**

```
1. Subscription Service: 구독 저장 + 이벤트 발행 (SUBSCRIPTION_CREATED)
2. Notification Service: 이벤트 수신 → WebSocket 세션 등록 + 응답 발행 (SESSION_REGISTERED)
3. Subscription Service: 응답 수신 → 구독 상태를 ACTIVE로 업데이트

실패 시 보상 트랜잭션:
2번 실패 → (REGISTRATION_FAILED) 이벤트 발행
→ Subscription Service: 구독 상태를 FAILED로 업데이트
```

#### 검증 (직접 실행해보기)

```bash
# Saga 흐름 추적 (Jaeger UI에서 TraceId로 확인)
# 또는 Kafka 토픽 메시지 순서 확인
kubectl exec -it kafka-controller-0 -n safealert-infra -- \
  kafka-console-consumer.sh --bootstrap-server localhost:9092 \
  --topic saga.subscription --from-beginning

# 구독 등록 후 DB 상태 확인
kubectl exec -it postgresql-0 -n safealert-infra -- psql -U postgres subscription_db
SELECT subscription_id, status FROM subscriptions ORDER BY created_at DESC LIMIT 5;
```

#### ✅ 이해 체크리스트

- [ ] Choreography와 Orchestration의 차이를 장단점 포함해 설명할 수 있다
- [ ] 이 프로젝트에서 Choreography를 선택한 근거를 말할 수 있다 (→ ADR-004 참고)
- [ ] 보상 트랜잭션(Compensating Transaction)이 무엇인지 구독 취소 예시로 설명할 수 있다

### Step 14: 의도적 장애 주입이 왜 포트폴리오에서 중요한가

#### 왜 이렇게 설계했나

면접관이 "장애를 경험해봤나요?"라고 물었을 때 "아직 없습니다"는 답변은 약하다.

```
의도적 장애 주입 (Chaos Engineering) 목적:
1. 시스템이 자동으로 복구되는지 직접 검증
2. 장애 → 감지 → 복구 과정을 경험하고 문서화
3. "저는 실제로 장애를 만들고 해결해봤습니다" 라는 구체적인 경험

검증할 장애 시나리오:
├── Pod 강제 종료 → K8s 자동 재시작 (30초 이내)
├── Kafka 브로커 중단 → Consumer 재연결, 메시지 유실 없음
├── Redis 접속 불가 → Fallback 동작
└── 공공 API Timeout → Circuit Breaker OPEN
```

#### 검증 (직접 실행해보기)

```bash
# 시나리오 1: Notification Service Pod 강제 종료
kubectl delete pod -n safealert-app -l app=notification-service --wait=false

# 실시간 복구 관찰
kubectl get pods -n safealert-app -w
# 기대: 30초 이내 새 Pod Running 상태

# Grafana에서 에러율 스파이크 후 정상화 확인
# Kafka Consumer Lag 확인 (일시 증가 후 0으로 수렴해야 함)
kafka-consumer-groups.sh --bootstrap-server ... --describe --group notification-group
```

**장애 테스트 결과 문서화 템플릿:**

```markdown
## 장애 시나리오: Notification Service Pod 종료

**날짜:** 2025-XX-XX
**발생:** kubectl delete pod notification-service-xxx
**감지:** Prometheus Alert - Pod 재시작 감지 (5초 이내)
**복구:** K8s ReplicaSet이 새 Pod 생성, 27초 후 Running
**결과:** Kafka Consumer Lag 최대 42개까지 증가 후 2분 내 0으로 복귀
**개선:** Liveness Probe 주기를 30초 → 10초로 단축
```

#### ✅ 이해 체크리스트

- [ ] Kubernetes ReplicaSet이 Pod 장애를 어떻게 감지하고 복구하는지 설명할 수 있다
- [ ] 장애 발생 시 Kafka Consumer Lag이 왜 증가하고 어떻게 줄어드는지 설명할 수 있다
- [ ] 각 장애 시나리오에 대한 복구 결과를 수치로 말할 수 있다 (면접 대비)

### Step 15: Prometheus가 Pull 방식으로 메트릭을 수집하는 이유

#### 왜 이렇게 설계했나

```
Push 방식 (각 서비스가 Prometheus에 데이터 전송)
└── 단점: 서비스가 Prometheus 주소를 알아야 함
└── 단점: Prometheus 주소가 바뀌면 모든 서비스 수정

Pull 방식 (Prometheus가 각 서비스에서 수집)
└── 장점: 서비스는 /actuator/prometheus 엔드포인트만 열면 됨
└── 장점: Prometheus가 스케줄에 따라 알아서 수집
└── 장점: 서비스 추가 시 ServiceMonitor만 추가하면 됨
```

**수집 흐름:**

```
각 서비스의 /actuator/prometheus → Prometheus가 15초마다 Pull
→ Prometheus 내부 시계열 DB 저장
→ Grafana가 Prometheus에서 쿼리하여 시각화
```

#### 검증 (직접 실행해보기)

```bash
# 서비스 메트릭 엔드포인트 직접 확인
curl http://auth-service:8081/actuator/prometheus | head -30

# Prometheus UI에서 쿼리
# http://localhost:9090
# 쿼리: http_server_requests_seconds_count{service="auth-service"}

# Grafana 접속
# http://localhost:3000 (admin/admin)
```

#### ✅ 이해 체크리스트

- [ ] Pull 방식과 Push 방식의 차이와 Pull을 선택한 이유를 설명할 수 있다
- [ ] `http_server_requests_seconds_count` 같은 메트릭 이름을 보고 무슨 데이터인지 해석할 수 있다
- [ ] Grafana가 직접 서비스에 접근하지 않고 Prometheus를 통하는 이유를 설명할 수 있다

### Step 16: 분산 트레이싱(Jaeger)이 왜 필요한가

#### 왜 이렇게 설계했나

```
단순 로그만 있는 경우 (문제)
사용자 A의 알림이 안 왔다는 신고
→ auth-service 로그 확인 → 이상 없음
→ subscription-service 로그 확인 → 이상 없음
→ alert-processor 로그 확인 → 이상 없음
→ notification-service 로그 확인 → ... 
→ 어느 서비스에서 문제가 생겼는지 파악 불가 (로그가 서비스별로 분산)

Jaeger 분산 트레이싱 (해결)
→ 하나의 요청에 TraceId 부여
→ 모든 서비스가 같은 TraceId로 로그 기록
→ Jaeger UI에서 TraceId 검색 → 전체 흐름 한 눈에 파악
→ 어느 서비스에서 지연/에러 발생했는지 즉시 확인
```

#### 검증 (직접 실행해보기)

```bash
# API 호출 후 응답 헤더에서 TraceId 확인
curl -v http://localhost:8080/api/subscriptions \

# Jaeger UI에서 TraceId 검색
# http://localhost:16686
# Service: api-gateway 선택 → Find Traces
# 특정 Trace 클릭 → 각 서비스별 소요 시간 확인
```

#### ✅ 이해 체크리스트

- [ ] TraceId와 SpanId의 차이를 설명할 수 있다 (Trace = 전체 요청, Span = 각 서비스 처리 단위)
- [ ] Kafka 메시지에도 TraceId를 전파해야 하는 이유를 설명할 수 있다
- [ ] Jaeger로 병목 구간을 찾는 방법을 실제로 시연할 수 있다

### Step 17: ELK Stack 각 컴포넌트 역할

#### 왜 이렇게 설계했나

```
Elasticsearch (E): 로그 저장소 + 검색 엔진
  → 구조화된 JSON 로그를 인덱싱해서 빠르게 검색 가능

Logstash (L): 로그 수집/파싱 파이프라인
  → 여러 소스의 로그를 수집, 필드 파싱, Elasticsearch로 전송

Kibana (K): 시각화 대시보드
  → Elasticsearch 데이터를 검색/시각화

Filebeat: 각 Pod의 로그를 수집해 Logstash로 전송하는 경량 에이전트
```

**로그 수집 흐름:**

```
각 서비스 Pod
  └── stdout 로그 (JSON 형식)

          └── Logstash
              └── Elasticsearch
                  └── Kibana
```

### Step 18: k6로 부하 테스트하는 방법과 결과 해석

#### 왜 이렇게 설계했나

```
실제 수백만 사용자가 없는 환경에서
"대규모 트래픽 처리 경험"을 증명하는 방법:

k6 부하 테스트 → Grafana에서 성능 측정 결과 캡처
→ "10,000 동시 사용자 기준 p95 응답시간 180ms" 같은 수치로 증명
```

#### 검증 (직접 실행해보기)

```javascript
// k6 스크립트 예시: 알림 API 부하 테스트
import http from 'k6/http';
import { check } from 'k6';

export const options = {
  stages: [
    { duration: '1m', target: 100 },   // 1분 동안 100 VU로 증가
    { duration: '3m', target: 1000 },  // 3분 동안 1000 VU 유지
    { duration: '1m', target: 0 },     // 1분 동안 0으로 감소
  ],
};

export default function () {
  const res = http.get('http://localhost:8080/api/notifications', {
    headers: { 'Authorization': `Bearer ${__ENV.TOKEN}` },
  });
  check(res, { 'status 200': (r) => r.status === 200 });
}
```

```bash
# 부하 테스트 실행
k6 run -e TOKEN={jwt_token} load-test.js

# 결과 해석
# http_req_duration p(95) < 200ms 목표
# http_req_failed 비율 < 1% 목표
# Grafana에서 동시 부하 중 CPU/메모리/Kafka lag 변화 관찰
```

#### ✅ 이해 체크리스트

- [ ] VU(Virtual User)가 무엇인지, stages 설정이 어떤 트래픽 패턴을 만드는지 설명할 수 있다
- [ ] p95, p99 응답시간이 평균 응답시간과 왜 다르고 어느 것이 더 중요한지 설명할 수 있다
- [ ] 부하 테스트 결과에서 병목이 DB인지, 서비스인지, Kafka인지 구분하는 방법을 설명할 수 있다
