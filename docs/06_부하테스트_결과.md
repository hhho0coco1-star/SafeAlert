## 1. 로그인 부하 테스트 (`login.js`)

### 시나리오

- **대상 API**: `POST /api/auth/login`
- **흐름**: 이메일 + 비밀번호 → JWT 토큰 발급
- **동시 사용자**: 최대 100명
- **지속 시간**: 50초 (10초 증가 → 30초 유지 → 10초 감소)

### 목표 기준 (Threshold)

| 지표 | 목표 |
|------|------|
| 에러율 | 1% 미만 |
| p(95) 응답시간 | 2초 이내 |

---

### 1차 테스트 — 개선 전 (HikariCP 기본값: 커넥션 10개)

| 지표 | 결과 | 목표 달성 |
|------|------|---------|
| 총 요청 수 | 1,475건 | - |
| 에러율 | **0.00%** | ✅ |
| 평균 응답시간 | 1.78s | - |
| p(90) 응답시간 | 2.51s | - |
| p(95) 응답시간 | **2.66s** | ❌ (목표 2초 초과) |
| 최대 응답시간 | 3.91s | - |

**문제 원인 분석**

```
100명이 동시에 로그인 요청
  → auth-service가 PostgreSQL에 사용자 정보 조회 (SELECT)
  → bcrypt로 비밀번호 해싱 검증 (CPU 집약적 연산)
  → HikariCP 커넥션 풀이 기본값 10개
  → 100명 중 90명이 커넥션 자리를 기다리며 대기 발생
  → 응답시간 지연
```

**개선 방법**: HikariCP `maximum-pool-size` 10 → 30 으로 증가

```yaml
# auth-service/src/main/resources/application.yml
spring:
  datasource:
    hikari:
      maximum-pool-size: 30
```

### 개선 전/후 비교

| 지표 | 개선 전 | 개선 후 | 개선율 |
|------|---------|---------|--------|
| 평균 응답시간 | 1.78s | **1.22s** | **31% 감소** |
| p(95) 응답시간 | 2.66s | **1.86s** | **30% 감소** |
| 에러율 | 0% | 0% | 유지 |
| Threshold 달성 | ❌ 실패 | ✅ 통과 | - |

> **결론**: DB 커넥션 풀 크기를 10 → 30으로 늘린 것만으로 p(95) 응답시간이 2.66s → 1.86s로 30% 개선됐다.
> 100명 동시 로그인 시 에러 없이 안정적으로 처리 가능함을 확인.
