## 1. 로그인 부하 테스트 (`login.js`)

### 시나리오

- **대상 API**: `POST /api/auth/login`
- **흐름**: 이메일 + 비밀번호 → JWT 토큰 발급
- **동시 사용자**: 최대 100명
- **지속 시간**: 50초 (10초 증가 → 30초 유지 → 10초 감소)

### 목표 기준 (Threshold)

| 지표 | 목표 |
|------|------|
| 에러율 | 1% 미만 |
| p(95) 응답시간 | 2초 이내 |

---

### 1차 테스트 — 개선 전 (HikariCP 기본값: 커넥션 10개)

| 지표 | 결과 | 목표 달성 |
|------|------|---------|
| 총 요청 수 | 1,475건 | - |
| 에러율 | **0.00%** | ✅ |
| 평균 응답시간 | 1.78s | - |
| p(90) 응답시간 | 2.51s | - |
| p(95) 응답시간 | **2.66s** | ❌ (목표 2초 초과) |
| 최대 응답시간 | 3.91s | - |

**문제 원인 분석**

```
100명이 동시에 로그인 요청
  → auth-service가 PostgreSQL에 사용자 정보 조회 (SELECT)
  → bcrypt로 비밀번호 해싱 검증 (CPU 집약적 연산)
  → HikariCP 커넥션 풀이 기본값 10개
  → 100명 중 90명이 커넥션 자리를 기다리며 대기 발생
  → 응답시간 지연
```

**개선 방법**: HikariCP `maximum-pool-size` 10 → 30 으로 증가

```yaml
# auth-service/src/main/resources/application.yml
spring:
  datasource:
    hikari:
      maximum-pool-size: 30
```

## 2. 구독 조회 부하 테스트 (`subscription.js`)

### 시나리오

- **대상 API**: `GET /api/subscriptions`
- **흐름**: 로그인으로 JWT 토큰 획득 → 토큰을 들고 구독 목록 조회
- **동시 사용자**: 최대 100명 (로그인 테스트와 동일 조건)
- **지속 시간**: 50초 (10초 증가 → 30초 유지 → 10초 감소)

### 목표 기준 (Threshold)

| 지표 | 목표 |
|------|------|
| 에러율 | 1% 미만 |
| p(95) 응답시간 | 1초 이내 |

---

### 테스트 결과

| 지표 | 결과 | 목표 달성 |
|------|------|---------|
| 총 요청 수 | 1,963건 | - |
| 에러율 | **0.00%** | ✅ |
| 평균 응답시간 | **89ms** | - |
| p(90) 응답시간 | 132ms | - |
| p(95) 응답시간 | **375ms** | ✅ (목표 1초 이내) |
| 최대 응답시간 | 2.15s | - |

**분석**

```
구독 조회는 단순 PostgreSQL SELECT 쿼리
  → bcrypt 연산 없음
  → 평균 89ms — 로그인(1.22s)보다 약 14배 빠름
  → 100명 동시 요청 시 에러 없이 안정적으로 처리
```

**로그인 vs 구독 조회 비교 (동일 100명 기준)**

| 지표 | 로그인 | 구독 조회 |
|------|--------|---------|
| 평균 응답시간 | 1.22s | **89ms** |
| p(95) 응답시간 | 1.86s | **375ms** |
| 에러율 | 0% | 0% |

> **결론**: 구독 조회 API는 100명 동시 요청에서도 p(95) 375ms로 안정적. 로그인 대비 5배 빠름. 추가 최적화 불필요.

## 3. WebSocket 연결 부하 테스트 (`websocket.js`)

### 시나리오

- **대상**: `ws://localhost:8080/api/notifications/ws`
- **흐름**: 로그인 토큰 획득 → JWT 인증 헤더로 WebSocket 연결 → STOMP CONNECT 전송
- **동시 연결**: 최대 50개
- **지속 시간**: 50초 (10초 증가 → 30초 유지 → 10초 감소)

### 목표 기준 (Threshold)

| 지표 | 목표 |
|------|------|
| ws_connecting p(95) | 3초 이내 |
| ws_session_duration p(95) | 35초 이내 |

---

### 테스트 결과

| 지표 | 결과 | 목표 달성 |
|------|------|---------|
| 총 WebSocket 세션 수 | 1,927개 | - |
| ws_connecting 평균 | **5.14ms** | - |
| ws_connecting p(95) | **9.91ms** | ✅ (목표 3초 대비 매우 빠름) |
| 에러 | 0개 | ✅ |

**분석**

```
50개 동시 WebSocket 연결
  → ws_connecting p(95) 9.91ms — 연결 수립 매우 빠름
  → 1,927개 세션 에러 없이 정상 처리
  → notification-service WebSocket 서버가 동시 연결 부하에 안정적으로 동작
```

> **결론**: 50개 동시 WebSocket 연결 시 p(95) 9.91ms로 연결 수립. 에러 0개로 안정적.
