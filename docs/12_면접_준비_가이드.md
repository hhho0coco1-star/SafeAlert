# SafeAlert 면접 준비 가이드

> 이 문서는 SafeAlert 프로젝트를 기반으로 한 기술 면접 대비용이다.
> 형식: **예상 질문 → 모범 답변 → (필요 시) 해설**
> 답변은 외워서 그대로 말하기보다, 핵심 키워드와 흐름을 이해해서 내 말로 재구성하는 것을 목표로 한다.

---

# 1부. 자기소개 / 프로젝트 소개 스크립트

## 1-1. 한 문장 요약 (엘리베이터 피치)

> "SafeAlert는 기상청·행정안전부·환경부 공공 API에서 재난 정보를 수집해, 사용자가 구독한 지역과 카테고리에 맞춰 **5초 이내 실시간으로 알림을 보내는** MSA 기반 시스템입니다. Kafka 이벤트 파이프라인, WebSocket 실시간 푸시, Kubernetes 배포까지 실제 프로덕션에 가까운 구조를 직접 구현했습니다."

## 1-2. 30초 버전

> "단순 CRUD를 넘어서 분산 시스템을 제대로 경험해보고 싶어서 만든 프로젝트입니다.
> 5개의 마이크로서비스와 API Gateway(프론트엔드 포함 K8s 7개 배포)로 구성했고, 공공 API에서 수집한 재난 정보가 Kafka를 거쳐 처리·분류된 뒤 WebSocket으로 사용자에게 실시간 전달됩니다.
> 메시지 유실 방지를 위해 Transactional Outbox 패턴을, 외부 API 장애 대응으로 Circuit Breaker를, 다중 인스턴스 환경의 WebSocket 동기화로 Redis Pub/Sub을 적용했습니다.
> 마지막으로 Prometheus·Grafana·Jaeger·ELK로 관측 가능성을 갖추고, k6로 부하 테스트, minikube로 K8s 배포와 HPA 오토스케일링까지 검증했습니다."

## 1-3. 3분 버전 (꼬리질문 유도용 구조)

1. **문제 정의** — 재난 알림은 "유실되면 안 되고", "빨라야 하고", "트래픽이 예측 불가능"하다. 이 세 가지 제약이 모든 기술 선택의 기준이었다.
2. **아키텍처** — 기능별로 배포·확장 요구가 달라 MSA로 분리. 수집(Collector) / 처리(Processor) / 발송(Notification)을 Kafka로 느슨하게 연결.
3. **핵심 설계** — ① 유실 방지: Outbox + Kafka 영속성, ② 속도: 이벤트 기반 + WebSocket Push, ③ 확장: Processor Replica 3 + Kafka 파티션 분산.
4. **안정성** — Circuit Breaker(외부 API), Fail-Open Rate Limiting(Redis), 장애 주입 테스트로 검증.
5. **운영** — 관측 가능성 스택 + 부하 테스트로 병목(HikariCP)을 찾아 개선, K8s/HPA로 마무리.

> 팁: 3분 버전은 "문제 → 선택 → 검증"의 흐름으로 말하면, 면접관이 자연스럽게 "그 부분 더 설명해주세요"로 들어온다. 내가 준비된 영역으로 대화를 유도하는 효과가 있다.

# 2부. 전체 동작 흐름

> "전체적으로 어떻게 동작하나요?" 는 거의 반드시 나온다. 아래 흐름을 막힘없이 말할 수 있어야 한다.

## 2-1. 재난 알림 발송 흐름 (이 프로젝트의 심장)

```
공공 API (기상청·행안부·환경부)
  │  ① Alert Collector가 1시간 주기로 폴링
  ▼
Alert Collector :8086
  │  ② Redis로 중복 체크(이미 수집한 알림인지) → 신규만 통과
  │  ③ Circuit Breaker로 외부 API 장애 격리
  ▼  alert.raw 토픽 (파티션 3)
Kafka
  ▼  ④ Consumer Group으로 3개 파티션 병렬 소비
Alert Processor :8087 (Replica 3)
  │  ⑤ 지역코드 파싱, 카테고리 분류, 중복 필터(Redis TTL 24h)
  │  ⑥ MongoDB(event_store)에 원본 저장
  ▼  alert.processed 토픽 (파티션 6)
Kafka
  ▼  ⑦ 소비
Notification Service :8083
  │  ⑧ Subscription Service에서 region+category 매칭 구독자 조회
  │  ⑨ notification_history(PostgreSQL)에 이력 저장
  │  ⑩ Redis Pub/Sub "alert:public" 채널에 발행 (다중 인스턴스 동기화)
  ▼  ⑪ WebSocket(STOMP) /topic/public/alerts 로 Push
브라우저 (실시간 수신)
```

**면접에서 강조할 포인트 (꼭 필요한 부분)**

- **왜 Collector와 Processor를 나눴나?** 수집은 외부 API에 종속(1시간 주기, 단일 인스턴스로 충분)이고, 처리는 트래픽 급증 시 확장이 필요(Replica 3). 책임과 확장 요구가 달라서 분리했다.
- **중복 체크가 두 번 나오는 이유** — Collector 단계(같은 재난을 또 수집하는 것 방지)와 Processor 단계(Kafka의 At-Least-Once 특성상 같은 메시지가 두 번 소비될 수 있어 멱등 처리)는 목적이 다르다.
- **Redis Pub/Sub이 ⑩에 들어간 이유** — WebSocket 세션은 서버 메모리에 저장되므로, 알림을 처리한 인스턴스와 사용자가 연결된 인스턴스가 다르면 푸시가 안 된다. (3부·5부에서 상세)

## 2-2. 인증이 필요한 API 요청 흐름

```
React → API Gateway
  │  ① Gateway가 JWT 서명·만료 검증 (각 서비스가 검증하지 않음)
  │  ② 유효하면 X-User-Id 헤더를 붙여 백엔드로 전달
  ▼
백엔드 서비스 (헤더의 userId만 신뢰하고 처리)
```

- **핵심**: 인증을 Gateway 한 곳에 집중(중앙 인증). 백엔드 서비스는 JWT 검증 로직 없이 `X-User-Id` 헤더만 읽으면 된다 → 코드 중복 제거, 시크릿 키 관리 일원화.
- **토큰 만료 시**: 401 반환 → 프론트가 Refresh Token으로 재발급 요청 → Gateway가 Redis에서 Refresh Token 유효성 확인 → 새 Access Token 발급.

## 2-3. 구독 등록 + Outbox 흐름

```
구독 추가 요청
  ▼
Subscription Service
  │  ① 한 트랜잭션 안에서: subscriptions 저장 + outbox_events 저장(PENDING)
  │  ② COMMIT (둘 다 성공 or 둘 다 실패 → 원자성 보장)
  ▼
  (별도) 스케줄러 5초 주기
  │  ③ outbox_events의 PENDING 조회 → Kafka 발행 → PUBLISHED로 변경
  ▼
Kafka
```

- **핵심**: "DB 저장"과 "Kafka 발행"은 하나의 트랜잭션으로 묶을 수 없다. 그래서 이벤트를 일단 같은 DB 트랜잭션 안의 Outbox 테이블에 적어두고, 별도 스케줄러가 나중에 Kafka로 보낸다. Kafka가 죽어 있어도 PENDING으로 남아 있다가 복구 후 발행된다.

# 3부. 기술 선택 이유 (가장 빈출)

> 면접관은 "왜 썼나"만큼 "**단점은 뭔가**(트레이드오프)"를 본다. 장점만 말하면 깊이가 없어 보인다. 항상 단점/대안과 함께 답한다.

## 3-1. 왜 MSA인가? (모놀리식 대비)

- **답변**: 기능별로 배포 주기와 확장 요구가 다르기 때문이다. Alert Processor는 재난 시 트래픽이 몰려 Replica 3으로 확장이 필요하지만, Auth는 변경이 드물고, Collector는 1시간 주기 단일 인스턴스면 충분하다. 모놀리식이면 Processor 하나 확장하려고 전체를 복제해야 한다.
- **트레이드오프(인정)**: 네트워크 통신 오버헤드, 분산 트랜잭션 복잡도(→ Outbox로 해결), 분산 디버깅 난이도(→ Jaeger로 보완), 로컬에서 여러 서비스 동시 실행 부담.
- **솔직한 한 줄**: "1인 포트폴리오에는 오버엔지니어링 측면이 있지만, MSA 기술(Kafka·Circuit Breaker·분산 추적)을 제대로 경험하는 것이 목표였습니다. 실제로 규모가 작은 Statistics·Admin은 독립 서비스로 만들지 않고 통합해 과설계를 줄였습니다."

## 3-2. 왜 Kafka인가? (RabbitMQ / Redis Pub/Sub 대비)

| 항목 | Kafka | RabbitMQ | Redis Pub/Sub |
|------|-------|----------|---------------|
| 메시지 영속성 | 디스크에 보존 | 소비 후 삭제 | 없음(구독자 없으면 유실) |
| 처리량 | 초당 수십만 | 초당 수만 | 매우 빠름 |
| 재처리 | offset으로 과거 재소비 | 어려움 | 불가 |
| 병렬 소비 | 파티션 단위 Consumer Group | Queue 경쟁 소비 | - |

- **답변**: 재난 알림은 **유실되면 안 된다**. Kafka는 디스크에 메시지를 보존하므로 브로커가 잠깐 죽었다 살아나도 offset부터 재소비할 수 있다. 또 Processor를 3개로 확장할 때 파티션 기반으로 자연스럽게 부하가 분산된다.
- **KRaft 모드**: Zookeeper 없이 단독 실행 → 로컬·K8s 구성 단순화.
- **트레이드오프**: 운영 복잡도가 높고 리소스 소모가 큼(minikube 메모리 8GB+). 소규모엔 오버스펙.
- **왜 Redis Pub/Sub은 메인 파이프라인에 안 썼나**: 영속성이 없어 구독자가 없으면 메시지가 사라진다. 그래서 Redis Pub/Sub은 WebSocket 인스턴스 동기화처럼 "실시간 전달만 되면 되는" 용도에만 쓰고, 유실되면 안 되는 알림 파이프라인엔 Kafka를 분리 적용했다.

## 3-3. 왜 Transactional Outbox 패턴인가?

- **문제**: DB 저장과 Kafka 발행을 한 트랜잭션으로 묶을 수 없다. "DB 저장 성공 → Kafka 발행 실패" 시 이벤트가 유실된다.
- **답변**: 비즈니스 데이터와 이벤트(Outbox)를 **같은 DB 트랜잭션**으로 저장하고, 별도 스케줄러가 Outbox를 폴링해 Kafka로 발행한다. Kafka가 죽어도 PENDING으로 남아 복구 후 재발행되어 **최종 일관성(Eventual Consistency)**을 보장한다.
- **트레이드오프**: At-Least-Once 발행이므로 같은 이벤트가 두 번 갈 수 있다 → Consumer가 `alertId`로 **멱등 처리** 필요. Outbox 테이블이 커지므로 PUBLISHED 이벤트 주기적 정리 필요.
- **왜 2PC(분산 트랜잭션) 안 썼나**: 강한 일관성을 주지만 성능 저하·복잡도가 커서 MSA에서 권장되지 않는다.

## 3-4. 왜 Redis인가? (3가지 용도)

1. **JWT Refresh Token 저장** (`token:refresh:{userId}`, TTL 7일) — DB 조회 없이 O(1) 검증, 로그아웃 시 즉시 삭제로 JWT의 "서버에서 못 끊는다" 단점 보완.
2. **Rate Limiting** (`ratelimit:{ip}`, TTL 1분) — 원자적 INCR로 동시성 문제 없이 분당 60건 초과 차단. DB 조회 없음.
3. **WebSocket 다중 인스턴스 동기화** (Pub/Sub) — 어느 인스턴스가 알림을 처리하든 Redis 채널에 발행하면 모든 인스턴스가 받아 자기 세션 사용자에게 푸시.

## 3-5. 왜 WebSocket + STOMP인가? (Polling / SSE 대비)

| 방식 | 특징 | 단점 |
|------|------|------|
| Polling | 구현 간단 | 불필요한 요청 다수, 지연 |
| SSE | 단방향 Push 특화, 간단 | 양방향 불가, 브라우저 연결 수 제한 |
| WebSocket | 양방향, 낮은 지연 | 구현 복잡, 세션 동기화 필요 |

- **답변**: 알림은 서버→클라이언트 Push가 핵심이고 지연이 적어야 한다. SSE도 가능하지만 추후 사용자 확인 응답 등 양방향 확장을 고려해 WebSocket을 선택. STOMP를 얹은 이유는 `/topic/...` 채널 기반 구독 관리가 편하고 Spring의 `@MessageMapping`으로 구현이 깔끔해서다.
- **트레이드오프**: 연결을 계속 유지하므로 동시 연결이 많아지면(예: 1만) 리소스 부하 → HPA로 인스턴스 확장 대응.

## 3-6. 왜 Circuit Breaker(Resilience4j)인가?

- **문제**: Collector가 외부 API 3종을 호출하는데, 한 API가 장애나면 타임아웃까지 스레드가 묶이고 연쇄 장애로 번질 수 있다.
- **답변**: 실패율 50% 초과 시 회로를 OPEN해 즉시 fallback(빈 결과)을 반환한다. 장애난 1개 API는 건너뛰고 나머지 2개는 정상 수집 → **단일 장애점 제거**. 30초 후 HALF-OPEN으로 재시도, 성공하면 CLOSED 복귀.

## 3-7. 왜 API Gateway(Spring Cloud Gateway)인가?

- **답변**: 인증을 각 서비스에 분산하면 코드 중복·시크릿 분산·정책 변경 시 전 서비스 수정이 발생한다. Gateway에서 JWT 검증 후 `X-User-Id`로 넘기면 서비스는 단순해진다. Rate Limiting도 한 곳에서 처리.
- **왜 Spring Cloud Gateway**: Spring 생태계 동일, WebFlux 기반 비동기 처리, 필터 체인 구성이 직관적.

## 3-8. 왜 PostgreSQL과 MongoDB를 같이 쓰나?

- **PostgreSQL**: 사용자·구독·알림 이력 등 **관계형 + 트랜잭션** 필요 데이터.
- **MongoDB**: 처리된 알림 원본(event_store). 공공 API 3종의 응답 구조가 다 다르고 바뀔 수 있어, 고정 스키마 대신 **JSON 문서를 그대로** 저장. 쓰기 중심이라 ACID 필요성 낮음.

## 3-9. 왜 JWT인가? (세션 대비)

- **답변**: MSA에서 세션을 쓰면 중앙 세션 저장소가 필요해 복잡해진다. JWT는 **자체 검증(self-contained)**이 가능해 Gateway에서 DB 조회 없이 검증된다.
- **이중 토큰**: Access(15분, 짧게 → 탈취 피해 최소화) + Refresh(7일, Redis 저장 → 로그아웃 즉시 무효화로 JWT 단점 보완).

## 3-10. 왜 Kubernetes인가? (Docker Compose 대비)

- **답변**: Replica 선언, 장애 시 자동 재시작, DNS 기반 서비스 디스커버리, HPA 오토스케일링은 Docker Compose가 못 한다. minikube와 실제 EKS의 API가 같아 실무 전환 비용이 작다.
- **트레이드오프**: 러닝커브·리소스 소모 큼. (그래서 일상 개발은 docker-compose+bootRun, 배포 검증만 minikube로 분리해 효율을 챙겼다.)

# 4부. CS 기초 지식 (면접 단골)

> 프로젝트와 연결해서 답하면 "아는 척"이 아니라 "써본 사람"으로 보인다. 각 항목에 SafeAlert 연결고리를 넣었다.

## 4-1. 네트워크

### HTTP vs HTTPS
- HTTP는 평문, HTTPS는 TLS로 암호화. 면접 포인트: **handshake**(클라이언트가 서버 인증서 검증 → 대칭키 교환 → 이후 대칭키로 암호화 통신), 대칭/비대칭 키 혼용 이유(비대칭은 안전하지만 느려서 키 교환만, 실제 데이터는 빠른 대칭키).

### WebSocket vs SSE vs Polling (★ 내 프로젝트 직결)
- **Polling**: 주기적으로 GET 요청. 간단하나 지연·낭비.
- **Long Polling**: 응답을 늦게 주는 방식으로 실시간 흉내.
- **SSE**: HTTP 연결 하나로 서버→클라 단방향 스트림. 자동 재연결 내장. 단방향만.
- **WebSocket**: 최초 HTTP 요청을 `Upgrade` 헤더로 핸드셰이크 후 **양방향 풀 듀플렉스** TCP 연결로 전환. SafeAlert는 이걸 선택.

### gRPC vs HTTP (★ Jaeger 트러블슈팅 직결)
- gRPC는 HTTP/2 기반(멀티플렉싱, 바이너리 protobuf) → 빠르지만 환경 설정 까다로움.
- HTTP/1.1은 느리지만 어디서나 안정적.
- 실제로 Jaeger 수집을 gRPC(4317)로 했다가 로컬 HTTP/2 문제로 실패 → HTTP/protobuf(4318)로 바꿔 해결한 경험이 있다.

### TCP vs UDP
- TCP: 연결지향, 순서·재전송 보장(신뢰성). HTTP·WebSocket·Kafka가 사용.
- UDP: 비연결, 빠르지만 유실 가능(DNS, 스트리밍, 게임).

### 로드밸런싱 / DNS
- K8s는 Service가 ClusterIP로 부하 분산, `service.namespace.svc.cluster.local` DNS로 서비스 디스커버리.

## 4-2. 데이터베이스

### 트랜잭션 / ACID (★ Outbox 직결)
- **A**tomicity(원자성), **C**onsistency(일관성), **I**solation(격리성), **D**urability(지속성).
- SafeAlert: 구독 저장 + Outbox 저장을 한 트랜잭션으로 묶어 원자성 확보.
- **격리 수준**: Read Uncommitted → Read Committed → Repeatable Read → Serializable. 부수 현상(Dirty/Non-repeatable/Phantom Read)과 함께 알아두기.

### 인덱스
- B-Tree 인덱스로 조회 속도 ↑, 쓰기 속도와 저장공간은 ↓ (트레이드오프). 카디널리티 높은 컬럼·WHERE/JOIN/ORDER BY에 사용되는 컬럼에 건다. 복합 인덱스는 **맨 왼쪽 컬럼부터** 사용해야 탄다.

### RDB vs NoSQL (★ Postgre+Mongo 직결)
- RDB: 정해진 스키마, JOIN, 강한 일관성/트랜잭션.
- NoSQL(MongoDB): 유연한 스키마, 수평 확장, 문서/키값. 일관성보다 가용성·확장성.
- SafeAlert는 데이터 성격에 따라 둘을 나눠 썼다(폴리글랏 퍼시스턴스).

### 정규화 / 반정규화
- 정규화: 중복 제거로 무결성↑, JOIN↑. 반정규화: 조회 성능 위해 의도적 중복. 통계 집계가 느리면 반정규화나 사전 집계(Read Model)를 고려.

### 커넥션 풀 (★ 부하 테스트 직결)
- DB 연결 생성 비용이 커서 미리 만들어 재사용. **HikariCP** 풀 크기 10→30으로 늘려 로그인 p95를 2.66s→1.86s로 30% 개선한 경험. 무작정 키우면 DB 부하·컨텍스트 스위칭으로 오히려 나빠질 수 있다(적정값 중요).

## 4-3. 운영체제 / 동시성

### 프로세스 vs 스레드
- 프로세스는 독립 메모리, 스레드는 메모리 공유(빠른 통신, 동기화 필요).
- 공유 자원 동시 접근 → race condition → 락/원자적 연산 필요. SafeAlert Rate Limiting의 Redis INCR가 원자적이라 락 없이 안전.

### 동기 vs 비동기 / 블로킹 vs 논블로킹
- API Gateway가 WebFlux(논블로킹) 기반인 이유: 적은 스레드로 많은 동시 연결 처리.

### 캐시
- 자주 읽는 데이터를 빠른 저장소(Redis)에 둬 응답 단축. **캐시 무효화 전략**(TTL, write-through 등)과 캐시 스탬피드 문제 알아두기.

## 4-4. 보안

### JWT vs 세션 (★ 직결)
- 세션: 서버가 상태 보관(stateful), 서버에서 강제 만료 쉬움, 확장 시 세션 공유 필요.
- JWT: stateless, 확장 유리, 단 발급 후 서버에서 강제 무효화가 어려움 → SafeAlert는 Refresh Token을 Redis에 둬서 보완.

### 비밀번호 해싱 (bcrypt)
- 단순 해시(SHA)는 빠르고 레인보우 테이블에 취약. bcrypt는 **salt + 의도적으로 느린** 연산(work factor)으로 무차별 대입을 어렵게 한다. SafeAlert 회원가입에 적용.

### OAuth2 (소셜 로그인)
- Authorization Code Grant 흐름: 사용자 → 인가 서버 로그인 → code 발급 → 서버가 code로 access token 교환 → 사용자 정보 조회. Google/Kakao 로그인에 적용.

### Rate Limiting
- IP/유저 단위 요청 제한으로 무차별 대입·DoS 완화. SafeAlert는 Redis로 분당 60건/IP.

## 4-5. 분산 시스템 (★ MSA 핵심)

### CAP 정리
- 분산 시스템은 **C**onsistency, **A**vailability, **P**artition tolerance 중 P가 사실상 필수라 C와 A 사이 트레이드오프. SafeAlert Rate Limiting은 Redis 장애 시 **Fail-Open**(가용성 우선)을 택했다.

### 멱등성(Idempotency)
- 같은 요청을 여러 번 처리해도 결과가 같음. Kafka At-Least-Once 때문에 같은 메시지가 두 번 와도 `alertId`로 중복 체크해 한 번만 처리.

### 최종 일관성(Eventual Consistency)
- 즉시 일관되진 않지만 시간이 지나면 일치. Outbox→Kafka 발행 사이 지연 동안은 불일치하나 결국 일치.

### Stateless 설계 (★ WebSocket 트러블슈팅 직결)
- 서버 메모리에 상태(세션·캐시)를 두면 다중 인스턴스에서 깨진다. 상태는 외부(Redis/DB)에 두고 서버는 요청만 처리. SafeAlert WebSocket 세션 문제를 Redis Pub/Sub로 해결한 근본 교훈이 이것.

### 분산 추적 (Distributed Tracing)
- 요청 하나가 여러 서비스를 거치면 어디서 느린지 알기 어렵다. traceId를 서비스·Kafka 메시지까지 전파해 Jaeger로 전체 흐름과 단계별 소요 시간을 시각화.

# 5부. 내 프로젝트 깊이 파고드는 질문 (꼬리질문 대비)

> "직접 했다"는 증거는 **구체적 수치와 실패 경험**이다. 아래 3개 스토리는 통째로 말할 수 있게 준비.

## 5-1. 트러블슈팅: WebSocket 다중 인스턴스 알림 미수신 (★ 대표 스토리)

- **상황**: Notification을 Replica 3으로 배포하니 간헐적으로 알림이 안 옴. 서버 1대일 땐 멀쩡했다.
- **원인**: WebSocket 세션은 서버 메모리에 저장된다. 사용자가 인스턴스 A에 연결됐는데 알림은 인스턴스 B가 처리하면, B는 그 사용자 세션을 몰라 푸시 불가.
- **해결**: Redis Pub/Sub 도입. 어느 인스턴스가 처리하든 `alert:public` 채널에 발행하면, 모든 인스턴스가 구독하다가 자기에게 연결된 사용자에게 푸시. (라디오 방송국 비유)
- **배운 점**: ① "서버 1대 통과 테스트가 다중 인스턴스에서 깨진다", ② 분산 환경 **Stateless 원칙**, ③ Redis Pub/Sub은 메시지를 저장하지 않으므로(구독자 있을 때만 전달) 알림 이력은 별도로 DB에 저장해야 한다.

## 5-2. 트러블슈팅: Jaeger 트레이스 수집 실패 (gRPC → HTTP)

- **상황**: OpenTelemetry 설정 후 Jaeger UI에 서비스가 안 뜸.
- **원인**: gRPC(4317, HTTP/2)로 설정했는데 로컬 docker-compose에서 HTTP/2가 제대로 동작 안 함 + Jaeger 쪽 gRPC 수신 미활성.
- **해결**: 엔드포인트를 HTTP/protobuf(4318)로 전환 → 6개 서비스 트레이스 정상 수집.
- **배운 점**: "설정했는데 안 될 때" 디버깅 순서 — ①컨테이너 로그 ②포트 확인 ③공식 문서 기본값 대조 ④단순한 방법(HTTP)으로 먼저 동작 확인 후 고급(gRPC) 시도.

## 5-3. 장애 주입 테스트 3종 (Chaos)

| 장애 | 시스템 반응 | 데이터 유실 |
|------|------------|------------|
| Kafka 브로커 중단 | 구독 저장은 성공, Outbox에 PENDING 누적 → 재시작 후 스케줄러가 재발행 | 없음 |
| Redis 중단 | Rate Limiting Fail-Open(요청 통과), 복구 시 자동 재활성화 | 없음 |
| 공공 API 오류 | Circuit Breaker OPEN → fallback, 나머지 2개 API는 정상 수집 | 없음 |

- **Circuit Breaker 숨은 버그 스토리**: 공공 API가 오류를 HTTP 200 + `{"resultCode":"03"}`으로 반환해 CB가 5xx만 보고 장애로 인식 못 함. → 응답 코드를 검사해 명시적으로 예외를 던지도록 수정하니 CB가 정상 작동. (HTTP 상태코드와 비즈니스 응답코드는 다르다는 교훈)

## 5-4. 부하 테스트 (k6) — 수치로 말하기

| 시나리오 | 동시 | 에러율 | p95 |
|---------|------|--------|-----|
| 로그인 | 100명 | 0% | 1.86s |
| 구독 조회 | 100명 | 0% | 375ms |
| WebSocket 연결 | 50개 | 0 | 9.91ms |

- **병목 발견·개선**: 로그인 p95가 높아 분석 → DB 커넥션 풀 부족이 원인. HikariCP 10→30으로 p95 2.66s→1.86s, **30% 개선**.
- **포인트**: "측정 → 병목 식별 → 개선 → 재측정"의 사이클을 돌렸다는 점이 핵심. 단순히 "테스트했다"가 아니라 수치로 개선했다.

## 5-5. K8s / 오토스케일링

- 7개 서비스 전부 minikube(AWS EC2 t3.xlarge)에 배포해 전 Pod Running 확인.
- **HPA**: api-gateway CPU 70% 임계값, 부하 발생 시 133%까지 올라 Pod가 2→4로 자동 확장되는 것을 실측.

# 6부. 압박/약점 질문 대비

> 가장 중요한 파트. 약점을 숨기지 말고 **인정 + 판단 근거 + 보완책**으로 답한다.

### Q. 1인 프로젝트인데 MSA는 오버엔지니어링 아닌가요?
> "맞습니다. 1인 규모에선 모놀리식이 생산성이 높습니다. 다만 이 프로젝트의 목적이 분산 시스템 기술을 제대로 경험하는 것이었고, 재난 알림이라는 도메인이 '유실 방지·실시간·예측 불가 트래픽'이라 MSA의 장점이 실제로 필요한 시나리오였습니다. 동시에 과설계를 막기 위해 규모가 작은 Statistics·Admin은 독립 서비스로 만들지 않고 통합했고, 복잡한 Saga도 Outbox로 충분하다고 판단해 의도적으로 스킵했습니다. 즉, 무조건 다 쪼갠 게 아니라 필요한 만큼만 분리했습니다."

### Q. 실제 사용자 트래픽을 받아본 적은 없잖아요?
> "프로덕션 실트래픽 경험은 없습니다. 대신 k6로 100 동시 사용자·WebSocket 50 연결을 시뮬레이션하고, 병목(커넥션 풀)을 찾아 30% 개선했습니다. 또 장애 주입 테스트로 Kafka·Redis·외부 API가 죽는 상황에서의 동작을 검증했습니다. 실트래픽은 못 받아봤지만 '측정하고 개선하는 방법론'은 경험했습니다."

### Q. Kafka가 정말 필요했나요? 그냥 REST로 호출하면 안 되나요?
> "동기 REST 호출이면 Notification이 죽었을 때 Processor도 영향을 받고, 트래픽이 몰리면 호출이 밀립니다. Kafka로 비동기·느슨하게 연결하면 소비자가 잠깐 죽어도 메시지가 보존되고, 소비자를 늘려 처리량을 조절할 수 있습니다. 재난 알림의 유실 금지 요구사항 때문에 영속성 있는 메시지 브로커가 필요했습니다."

### Q. 테스트 코드는 충분히 작성했나요?
> "정직하게 말씀드리면 단위 테스트 커버리지는 높지 않습니다. **Auth Service의 핵심 비즈니스 로직은 JUnit5 + Mockito로 단위 테스트했습니다** — 회원가입 중복 이메일 예외, 로그인 성공/비밀번호 불일치, 비밀번호 재설정의 토큰 만료·재사용 방지·존재하지 않는 이메일/OAuth 계정 무시까지 7개 케이스입니다. 나머지 서비스는 Spring `contextLoads` 수준이고, 통합 동작은 **k6 부하 테스트와 장애 주입 테스트로 검증**했습니다. 단위 테스트가 부족한 건 인지하고 있고, 다음 우선순위는 Alert Processor의 멱등 중복 필터와 Subscription의 구독자 매칭 로직입니다."

> **실제 테스트 현황 (사실 기준 — 답변 시 이 범위 내에서 말할 것)**
> | 서비스 | 테스트 |
> |--------|--------|
> | Auth Service | ✅ `AuthServiceTest` — Mockito 단위 테스트 7케이스 (회원가입/로그인/비밀번호 재설정) |
> | Alert Collector | ⚠️ `contextLoads`만 (스프링 기본 생성 테스트) |
> | Alert Processor | ⚠️ `contextLoads`만 |
> | API Gateway / Subscription / Notification | ❌ 테스트 없음 |
> | Frontend | ❌ 테스트 없음 |
> | 통합/성능 | ✅ k6 부하 테스트 3종(login·subscription·websocket) + 장애 주입 테스트 3종 |
>
> ⚠️ 면접에서 "모든 서비스를 테스트했다"고 말하면 안 된다. Auth만 진짜 단위 테스트가 있다는 사실을 정확히 알고 답할 것.

### Q. 보안은 어떻게 고려했나요?
> "JWT 이중 토큰(Access 짧게+Refresh Redis로 무효화 가능), bcrypt 비밀번호 해싱, Gateway 중앙 인증, IP Rate Limiting을 적용했습니다. 다만 Rate Limiting은 Redis 장애 시 Fail-Open(가용성 우선)으로 두어 보안보다 가용성을 택한 트레이드오프가 있고, 프로덕션이라면 Fail-Closed도 검토할 수 있습니다."

### Q. 이 프로젝트에서 가장 어려웠던 점은?
> 5-1(WebSocket 다중 인스턴스) 스토리로 답한다. "단일 서버에선 안 보이던 문제가 분산 환경에서 드러났고, Stateless 설계라는 분산 시스템의 근본 원칙을 몸으로 배운 경험"이라고 마무리.

### Q. 다시 만든다면 무엇을 바꾸겠어요?
