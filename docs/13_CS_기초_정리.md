# SafeAlert 면접용 CS 기초 정리

> 이 문서는 **CS(컴퓨터 과학) 기초가 부족한 상태에서 처음부터** 정리하는 학습 자료다.
> 면접 가이드(12번)의 "4부 CS"가 요약본이라면, 이 문서는 **개념을 처음 보는 사람도 이해하도록 풀어 쓴 본편**이다.
> 구성: 각 항목마다 **① 개념(쉬운 설명) → ② 왜 중요한가 → ③ 면접 단골 질문 → ④ SafeAlert 연결고리** 순서.
>
> 공부 순서 추천: **Part 1(네트워크) → Part 4(DB) → Part 5(보안) → Part 2(OS) → Part 6(분산) → Part 3(자료구조)**
> (내 프로젝트와 연결이 강한 순서. 면접 빈도도 이 순서와 비슷하다.)

# Part 1. 네트워크

## 1-1. 클라이언트와 서버, 그리고 요청-응답

- **개념**: 브라우저(클라이언트)가 "이 데이터 주세요"라고 **요청(Request)**하면, 서버가 처리해서 **응답(Response)**한다. 인터넷의 거의 모든 통신이 이 구조다.
- **비유**: 식당에서 손님(클라이언트)이 주문(요청)하면 주방(서버)이 음식(응답)을 내준다.
- **면접 포인트**: "URL을 입력하면 화면이 뜨기까지 무슨 일이 일어나는가?" → ① DNS로 도메인→IP 변환 → ② TCP 연결(3-way handshake) → ③ (HTTPS면) TLS 핸드셰이크 → ④ HTTP 요청 전송 → ⑤ 서버 응답 → ⑥ 브라우저 렌더링.

## 1-2. HTTP / HTTPS

- **HTTP**: 웹에서 데이터를 주고받는 약속(프로토콜). 요청에는 **메서드**가 있다.
  - `GET`(조회), `POST`(생성), `PUT`(전체 수정), `PATCH`(부분 수정), `DELETE`(삭제).
- **상태 코드(Status Code)**: 응답의 결과를 숫자로 표현.
  - `2xx` 성공(200 OK, 201 Created), `3xx` 리다이렉트, `4xx` 클라이언트 잘못(400 Bad Request, 401 인증 안 됨, 403 권한 없음, 404 없음), `5xx` 서버 잘못(500, 503).
- **HTTPS**: HTTP에 **암호화(TLS)**를 입힌 것. 중간에서 누가 훔쳐봐도 내용을 못 본다.
  - **핵심 원리(혼합 암호화)**: 비대칭키(공개키/개인키)는 안전하지만 느려서 **"대칭키를 안전하게 교환"**하는 데만 쓰고, 실제 데이터는 빠른 **대칭키**로 암호화한다.
- **면접 단골**:
  - "HTTP와 HTTPS 차이?" → 암호화 유무, HTTPS는 TLS 핸드셰이크로 인증서 검증 + 키 교환.
  - "401과 403 차이?" → 401은 "너 누구인지 모름(인증 실패)", 403은 "누군지는 아는데 권한 없음(인가 실패)".
- **SafeAlert 연결**: API Gateway가 JWT로 인증 실패 시 **401**을 반환하고, 관리자 전용 API에 일반 사용자가 접근하면 **403** 성격의 처리를 한다.

## 1-3. HTTP 버전 (1.1 vs 2 vs 3) — gRPC 이해의 기반

- **HTTP/1.1**: 한 연결에서 요청을 하나씩 순서대로 처리(앞 요청이 느리면 뒤가 막힘 = HOL 블로킹).
- **HTTP/2**: 한 연결에서 여러 요청을 동시에(멀티플렉싱), 바이너리로 전송 → 빠름.
- **gRPC**: 구글이 만든 통신 방식. **HTTP/2 기반 + Protobuf(바이너리 직렬화)**라 빠르지만, 환경(프록시·로컬 설정)에 따라 까다롭다.
- **면접 단골**: "gRPC와 REST 차이?" → REST는 보통 HTTP/1.1 + JSON(읽기 쉬움, 범용), gRPC는 HTTP/2 + Protobuf(빠름, 서비스 간 내부 통신에 유리, 사람이 읽기 어려움).
- **SafeAlert 연결(실경험)**: Jaeger로 트레이스를 보낼 때 gRPC(4317)가 로컬 docker-compose에서 동작이 불안정 → **HTTP/protobuf(4318)로 전환**해 해결. "빠른 방식(gRPC)이 항상 정답은 아니고, 환경에 따라 안정적인 HTTP를 택할 수 있다"는 판단 경험.

## 1-4. TCP vs UDP

| | TCP | UDP |
|---|-----|-----|
| 연결 | 연결을 맺고 시작(3-way handshake) | 연결 없이 그냥 보냄 |
| 신뢰성 | 순서 보장 + 빠진 것 재전송 | 보장 안 함(유실 가능) |
| 속도 | 상대적으로 느림 | 빠름 |
| 용도 | 웹(HTTP), 메일, 파일전송 | 영상 스트리밍, 게임, DNS |

- **비유**: TCP는 "등기우편(받았는지 확인)", UDP는 "전단지 뿌리기(빠르지만 누가 받았는지 모름)".
- **면접 단골**: "실시간 게임에 왜 UDP를?" → 약간 유실돼도 최신 데이터가 빨리 오는 게 더 중요해서. 웹/금융은 정확성이 중요해 TCP.

## 1-5. 실시간 통신: Polling vs SSE vs WebSocket (★ SafeAlert 핵심)

서버가 클라이언트에게 **먼저** 데이터를 보내야 할 때(예: 재난 알림) 쓰는 방식들.

| 방식 | 동작 | 장점 | 단점 |
|------|------|------|------|
| **Polling** | 클라가 주기적으로 "새 거 있어?" 계속 물음 | 구현 단순 | 불필요한 요청 많음, 지연 |
| **Long Polling** | 물어보면 서버가 새 게 생길 때까지 응답을 미룸 | 실시간 흉내 | 연결 자원 소모 |
| **SSE** | HTTP 연결 하나로 서버→클라 **단방향** 스트림 | 간단, 자동 재연결 | 단방향만, 연결 수 제한 |
| **WebSocket** | 최초 HTTP로 시작 → `Upgrade`로 **양방향 상시 연결** 전환 | 양방향, 낮은 지연 | 구현 복잡, 연결 유지 비용 |

- **면접 단골**: "왜 WebSocket을 썼나? SSE는 안 됐나?" → 재난 알림은 단방향 Push가 핵심이라 SSE도 가능하지만, 추후 양방향 확장(사용자 확인 응답 등)을 고려해 WebSocket 선택. **STOMP**(WebSocket 위에 얹는 메시지 규약)로 `/topic/...` 채널 구독 구조를 깔끔하게 구현.
- **SafeAlert 연결**: 처리된 알림을 Notification Service가 WebSocket(STOMP)으로 브라우저에 푸시.

## 1-6. DNS / 로드밸런싱 / 서비스 디스커버리

- **DNS**: 도메인 이름(`google.com`)을 IP 주소로 바꿔주는 "전화번호부".
- **로드밸런서**: 요청을 여러 서버에 골고루 분배해 부하를 나눔.
- **면접 단골**: "MSA에서 서비스끼리 IP를 어떻게 찾나?" → IP를 하드코딩하지 않고 **DNS 기반 서비스 디스커버리** 사용.
- **SafeAlert 연결**: K8s에서 `서비스명.네임스페이스.svc.cluster.local` DNS로 서로를 찾고, Service가 Pod들에 부하를 분산.

# Part 2. 운영체제(OS)

## 2-1. 프로세스 vs 스레드

- **프로세스**: 실행 중인 프로그램 하나. **독립된 메모리 공간**을 가짐.
- **스레드**: 한 프로세스 안에서 동시에 실행되는 작업 흐름. **메모리를 공유**함.
- **비유**: 프로세스 = 식당 한 곳(독립 주방), 스레드 = 그 주방의 요리사 여러 명(같은 냉장고 공유).
- **면접 단골**:
  - "프로세스와 스레드 차이?" → 메모리 공유 여부. 스레드는 통신이 빠르지만 한 스레드가 잘못되면 전체에 영향, 공유 자원 동기화 필요.
  - "멀티프로세스 vs 멀티스레드?" → 안정성(프로세스) vs 효율성/속도(스레드)의 트레이드오프.

## 2-2. 동시성 문제와 동기화

- **경쟁 상태(Race Condition)**: 여러 스레드가 같은 데이터를 동시에 바꾸려다 결과가 꼬이는 현상.
  - 예: 잔액 1000원에서 두 요청이 동시에 500원씩 출금 → 둘 다 "1000원 확인" 후 출금 → 잔액이 잘못됨.
- **해결**: **락(Lock)**으로 한 번에 하나만 접근하게 하거나, **원자적 연산(Atomic)**을 사용.
- **SafeAlert 연결**: Rate Limiting에서 Redis의 **`INCR`(원자적 증가)**를 쓴다. 동시에 여러 요청이 와도 카운트가 꼬이지 않아 별도 락이 필요 없다.

## 2-3. 동기/비동기, 블로킹/논블로킹

- **동기(Sync)**: 일을 시키고 끝날 때까지 **기다림**.
- **비동기(Async)**: 일을 시켜놓고 **다른 일을 함**, 끝나면 알림 받음.
- **블로킹**: 결과가 올 때까지 스레드가 **멈춤**. **논블로킹**: 안 멈추고 진행.
- **면접 단골**: "비동기를 왜 쓰나?" → 적은 스레드로 많은 요청을 처리(자원 효율), 응답성 향상.
- **SafeAlert 연결**: API Gateway가 **Spring WebFlux(논블로킹)** 기반. 적은 스레드로 많은 동시 연결을 처리하기 위함. 또한 Kafka로 서비스를 **비동기**로 연결해 한 서비스가 느려도 전체가 안 막히게 함.

## 2-4. 메모리: 스택 vs 힙, 가비지 컬렉션(GC)

- **스택(Stack)**: 함수 호출·지역 변수가 쌓이는 곳. 빠르고 자동 정리.
- **힙(Heap)**: `new`로 만든 객체가 사는 곳. 크고 유연하지만 정리가 필요.
- **GC(가비지 컬렉션)**: 자바 등에서 **안 쓰는 객체를 자동으로 치워주는** 기능. 개발자가 메모리를 직접 해제 안 해도 됨.
- **면접 단골**: "메모리 누수(leak)란?" → 안 쓰는데 참조가 남아 GC가 못 치우는 객체가 쌓여 메모리가 고갈되는 것.
- **SafeAlert 연결**: 모니터링에서 **JVM 메모리/GC 메트릭**(Prometheus)을 수집해 메모리 상태를 관찰.

## 2-5. 캐시(Cache)

- **개념**: 자주 쓰는 데이터를 **더 빠른 곳**에 미리 둬서 응답을 빠르게.
- **면접 단골**:
  - "캐시 적중률(Hit Rate)?" → 요청 중 캐시에서 바로 답한 비율. 높을수록 좋음.
  - "캐시 무효화(Invalidation)?" → 원본이 바뀌면 캐시도 갱신해야 함(TTL 만료, write-through 등). "캐시 무효화는 컴퓨터 과학에서 가장 어려운 문제 중 하나"라는 말이 있을 정도.
- **SafeAlert 연결**: Redis를 캐시·세션 저장소로 사용. 알림 캐시(`alert:cache:{regionCode}`, TTL 5분) 등.

# Part 3. 자료구조 / 알고리즘 (기초)

## 3-1. 시간복잡도 (Big-O)

- **개념**: 데이터가 늘어날 때 처리 시간이 **얼마나 빨리 증가하는지**를 표기.
- **자주 나오는 순서(빠름→느림)**: O(1) < O(log n) < O(n) < O(n log n) < O(n²)
  - O(1): 입력과 무관하게 일정 (예: 배열 인덱스 접근, 해시맵 조회)
  - O(log n): 절반씩 줄임 (예: 이진 탐색, B-Tree 인덱스)
  - O(n): 전부 한 번씩 (예: 반복문 한 번)
  - O(n²): 이중 반복문
- **면접 단골**: "이 코드의 시간복잡도는?" → 반복문 중첩 수를 보면 대략 감이 옴.
- **SafeAlert 연결**: Redis 조회/Rate Limit이 **O(1)**이라 DB 조회 없이 빠르게 처리한다고 설명 가능.

## 3-2. 핵심 자료구조

| 자료구조 | 특징 | 조회 | 용도 |
|---------|------|------|------|
| 배열(Array) | 연속 메모리, 인덱스 접근 | O(1) | 순서 있는 목록 |
| 연결 리스트 | 노드를 포인터로 연결 | O(n) | 삽입/삭제 잦을 때 |
| 해시맵(HashMap) | key→value, 해시 함수 | O(1) 평균 | 빠른 조회 |
| 스택(Stack) | 후입선출(LIFO) | - | 되돌리기, 함수 호출 |
| 큐(Queue) | 선입선출(FIFO) | - | 작업 대기열, 메시지 |
| 트리(Tree) | 계층 구조 | O(log n) | 인덱스, 정렬 |

- **면접 단골**: "HashMap이 빠른 이유?" → 해시 함수로 위치를 바로 계산해 O(1). 단, 해시 충돌이 많으면 느려질 수 있음.
- **SafeAlert 연결**: Kafka는 본질적으로 **큐(메시지 대기열)** 개념. 메시지가 들어온 순서대로 소비된다(파티션 내에서).

## 3-3. 면접에서 코딩테스트로 자주 나오는 것

- 정렬(버블/퀵/병합 — 이름과 O 정도는 알기), 이진 탐색, 해시맵을 이용한 중복 제거/카운팅, 스택/큐 활용, BFS/DFS(그래프 탐색) 기초.
- **초보 팁**: 자료구조 외우기보다 "**언제 어떤 걸 쓰는지**"를 아는 게 면접에서 더 유용하다.

# Part 4. 데이터베이스 (DB)

## 4-1. RDB vs NoSQL (★ SafeAlert 핵심)

| | RDB (PostgreSQL, MySQL) | NoSQL (MongoDB, Redis) |
|---|---|---|
| 구조 | 표(테이블), 정해진 스키마 | 문서/키값 등, 유연한 스키마 |
| 관계 | JOIN으로 테이블 연결 | JOIN 약함, 보통 한 문서에 묶음 |
| 일관성 | 강함(ACID 트랜잭션) | 보통 가용성/확장성 우선 |
| 확장 | 수직 확장(서버 강화) 위주 | 수평 확장(서버 추가) 유리 |

- **면접 단골**: "RDB와 NoSQL 언제 쓰나?" → 관계와 정확성이 중요하면 RDB, 스키마가 자주 바뀌거나 대량/유연한 데이터면 NoSQL.
- **SafeAlert 연결(폴리글랏 퍼시스턴스)**: 사용자·구독·이력처럼 관계+트랜잭션이 필요한 데이터는 **PostgreSQL**, 공공 API 3종의 제각각인 원본 응답은 **MongoDB**(JSON 문서 그대로 저장)에 보관.

## 4-2. 트랜잭션과 ACID (★ Outbox 패턴의 기반)

- **트랜잭션**: 여러 작업을 **하나의 묶음**으로 처리. 전부 성공하거나 전부 실패(중간 상태 없음).
  - 예: 계좌 이체 = "A에서 출금 + B에 입금"을 한 트랜잭션으로 → 하나만 되면 안 됨.
- **ACID**:
  - **A**tomicity(원자성): 전부 or 전무.
  - **C**onsistency(일관성): 규칙(제약조건)을 항상 만족.
  - **I**solation(격리성): 동시에 실행돼도 서로 간섭 안 함.
  - **D**urability(지속성): 커밋되면 장애가 나도 보존.
- **면접 단골**: "ACID 설명해보세요" / "트랜잭션 격리 수준?"
  - 격리 수준(낮음→높음): Read Uncommitted → Read Committed → Repeatable Read → Serializable.
  - 부작용: **Dirty Read**(커밋 안 된 값 읽음), **Non-repeatable Read**(같은 조회가 값이 바뀜), **Phantom Read**(없던 행이 생김).
- **SafeAlert 연결**: 구독 저장과 Outbox 이벤트 저장을 **한 트랜잭션**으로 묶어 원자성 확보 → "DB는 저장됐는데 이벤트는 유실" 문제를 방지.

## 4-3. 인덱스(Index)

- **개념**: 책의 "찾아보기"처럼, 특정 컬럼을 빠르게 찾도록 만든 별도 자료구조(보통 **B-Tree**).
- **장점**: 조회(WHERE, JOIN, ORDER BY)가 빨라짐 (O(n) → O(log n)).
- **단점(트레이드오프)**: 저장 공간 추가, **쓰기(INSERT/UPDATE)가 느려짐**(인덱스도 갱신해야 하므로).
- **면접 단골**:
  - "인덱스를 무조건 많이 걸면 좋은가?" → 아니다. 쓰기 성능 저하 + 공간 낭비. 조회가 잦은 컬럼에 선별적으로.
  - "복합 인덱스 (a, b) 주의점?" → **맨 왼쪽 컬럼(a)부터** 조건에 써야 인덱스를 탄다.
  - "카디널리티?" → 값의 다양성. 높은 컬럼(예: 이메일)이 인덱스 효과가 크다.

## 4-4. 정규화 / 반정규화

- **정규화**: 중복을 제거해 데이터를 나눔 → 무결성↑, 하지만 조회 시 JOIN 많아짐.
- **반정규화**: 일부러 중복을 둠 → 조회 빠름, 하지만 갱신 시 일관성 관리 부담.
- **면접 단골**: "통계 조회가 느린데?" → 매번 원본을 집계하지 말고 **반정규화/사전 집계(미리 계산해 저장)**를 고려.
- **SafeAlert 연결**: 통계는 별도 Read 모델(CQRS)을 만들지 않고 Notification Service가 이력 테이블을 직접 집계 → 데이터 규모상 사전 집계의 실익이 작다고 판단(과설계 방지).

## 4-5. 커넥션 풀(Connection Pool) (★ 부하 테스트 실경험)

- **개념**: DB 연결을 만드는 비용이 크기 때문에, **미리 여러 개 만들어 두고 재사용**하는 기법.
- **면접 단골**: "풀 크기를 크게 하면 무조건 빠른가?" → 아니다. 너무 크면 DB 부하·컨텍스트 스위칭으로 오히려 느려진다. **적정값**이 중요.
- **SafeAlert 연결(수치로 말하기)**: 로그인 p95가 높아 분석 → DB 커넥션 풀 부족이 원인. **HikariCP 풀 10→30**으로 조정해 p95 **2.66s→1.86s (30% 개선)**. "측정 → 병목 식별 → 개선 → 재측정" 사이클 경험.

# Part 5. 보안 / 인증 (★ SafeAlert 핵심)

## 5-1. 인증(Authentication) vs 인가(Authorization)

- **인증**: "너 **누구**냐?" (로그인으로 신원 확인)
- **인가**: "너 이거 **해도 되냐**?" (권한 확인, 예: 관리자만 접근)
- **면접 단골**: 둘을 헷갈리면 감점. 401(인증 실패) vs 403(인가 실패)와 연결.

## 5-2. 세션 vs JWT (★ 매우 자주 나옴)

| | 세션(Session) | JWT(토큰) |
|---|---|---|
| 상태 | 서버가 기억(stateful) | 토큰 자체에 정보(stateless) |
| 저장 | 서버 메모리/DB에 세션 저장 | 클라이언트가 토큰 보관 |
| 확장성 | 서버 늘리면 세션 공유 필요 | 서버가 토큰만 검증하면 됨(유리) |
| 강제 만료 | 서버에서 쉽게 삭제 | 발급 후 강제 무효화 어려움 |

- **JWT 구조**: `헤더.페이로드.서명` 3부분. 서명으로 **위변조를 검증**(내용은 암호화가 아니라 인코딩이라 누구나 디코딩 가능 → 민감정보 넣으면 안 됨).
- **면접 단골**:
  - "JWT 단점과 보완?" → 발급 후 서버에서 못 끊는다 → **Refresh Token을 Redis에 저장**해 로그아웃/탈취 시 무효화로 보완.
  - "Access/Refresh 토큰을 왜 나누나?" → Access는 짧게(탈취 피해 최소화), Refresh는 길게(재로그인 줄임) + 무효화 가능.
- **SafeAlert 연결**: MSA라 세션 공유가 복잡 → JWT 선택. Access(15분) + Refresh(7일, Redis 저장). Gateway가 DB 조회 없이 토큰 검증.

## 5-3. 비밀번호 저장: 해싱과 솔트

- **절대 평문 저장 금지**. 단방향 **해시**로 저장(원문 복원 불가).
- **bcrypt**: 비밀번호 해싱 전용. **솔트(랜덤 값)**를 섞어 같은 비밀번호도 다르게 저장되고, **의도적으로 느리게**(work factor) 만들어 무차별 대입을 어렵게 함.
- **면접 단골**: "SHA-256으로 비밀번호 저장하면 안 되나?" → SHA는 너무 빨라 무차별 대입에 취약하고 솔트가 기본 없음. bcrypt/scrypt/argon2 권장.
- **SafeAlert 연결**: 회원가입 시 **bcrypt**로 해싱 저장(시퀀스 다이어그램에 명시). 비밀번호 재설정도 토큰 1회성·재사용 방지 로직 구현.

## 5-4. OAuth2 (소셜 로그인)

- **개념**: 내 서비스가 비밀번호를 직접 받지 않고, 구글/카카오에 **인증을 위임**받는 방식.
- **Authorization Code 흐름**: 사용자가 구글 로그인 → 구글이 **code** 발급 → 내 서버가 code를 **access token**으로 교환 → 그 토큰으로 사용자 정보 조회.
- **면접 단골**: "OAuth를 왜 쓰나?" → 비밀번호를 직접 보관 안 해 보안 부담↓, 가입 편의↑.
- **SafeAlert 연결**: Google/Kakao 소셜 로그인 구현.

## 5-5. 흔한 웹 취약점 (이름과 한 줄 방어책)

- **SQL Injection**: 입력으로 SQL을 조작 → **파라미터 바인딩(PreparedStatement)/ORM**으로 방어.
- **XSS**: 악성 스크립트 삽입 → 출력 시 **이스케이프**, 입력 검증.
- **CSRF**: 의도치 않은 요청 위조 → **CSRF 토큰**, SameSite 쿠키.
- **Rate Limiting**: 무차별 대입·DoS 완화 → 요청 횟수 제한.
- **SafeAlert 연결**: API Gateway에서 **IP당 분당 60건 Rate Limiting**(Redis). Redis 장애 시 **Fail-Open**(가용성 우선) 트레이드오프 적용.

# Part 6. 분산 시스템 / 아키텍처 (★ MSA 프로젝트의 핵심 CS)

## 6-1. 모놀리식 vs MSA

- **모놀리식**: 하나의 큰 애플리케이션. 개발·배포 단순, 디버깅 쉬움. 단, 일부만 확장 불가, 한 곳 장애가 전체 영향.
- **MSA(마이크로서비스)**: 기능별로 작은 서비스로 분리. 독립 배포·확장·장애 격리 가능. 단, 네트워크 통신·분산 트랜잭션·운영 복잡도↑.
- **면접 단골**: "MSA가 무조건 좋은가?" → 아니다. 작은 팀/프로젝트엔 오버엔지니어링. 조직·트래픽 규모가 커질 때 진가.
- **SafeAlert 연결**: 기능별 확장 요구가 달라 MSA 선택. 단, 규모가 작은 Statistics·Admin은 **통합**해 과설계를 줄임(필요한 만큼만 분리).

## 6-2. CAP 정리

- **개념**: 분산 시스템은 **C(일관성)·A(가용성)·P(분할 내성)** 셋 중 셋을 동시에 완벽히 가질 수 없다. 네트워크 장애(P)는 사실상 피할 수 없으므로 **C와 A 사이에서 선택**하게 된다.
  - **CP**: 일관성 우선(틀린 답 줄 바엔 응답 거부).
  - **AP**: 가용성 우선(조금 옛 데이터라도 응답).
- **면접 단골**: "CAP에서 무엇을 택했나?" → 시스템 성격에 따라. 금융=CP, SNS 피드=AP 경향.
- **SafeAlert 연결**: Rate Limiting의 Redis 장애 시 **Fail-Open(가용성 우선=A)** 선택 → 보호 기능보다 서비스가 죽지 않는 것을 우선.

## 6-3. 메시지 큐 / 이벤트 기반 아키텍처 (★ Kafka)

- **개념**: 서비스가 서로 직접 호출하지 않고, **메시지(이벤트)**를 큐에 넣고 빼며 **느슨하게(decoupled)** 연결.
- **장점**: 한 서비스가 죽어도 메시지가 보존됨, 소비자를 늘려 처리량 조절, 비동기로 응답 빠름.
- **Kafka 핵심 용어**:
  - **토픽(Topic)**: 메시지 분류 채널. **파티션(Partition)**: 토픽을 쪼갠 단위(병렬 처리의 키).
  - **프로듀서/컨슈머**: 발행자/소비자. **컨슈머 그룹**: 같은 그룹의 여러 인스턴스가 파티션을 나눠 병렬 소비.
  - **오프셋(Offset)**: 어디까지 읽었는지 위치. 덕분에 **재처리** 가능.
- **면접 단골**: "Kafka vs RabbitMQ?" → Kafka는 디스크 영속성·고처리량·재처리(로그 기반), RabbitMQ는 유연한 라우팅·소비 후 삭제(전통 큐).
- **SafeAlert 연결**: 재난 알림은 **유실 금지** → 영속성 있는 Kafka. Processor를 3개 인스턴스로 두고 파티션으로 부하 분산.

## 6-4. 멱등성(Idempotency)

- **개념**: 같은 요청을 **여러 번 해도 결과가 같음**. (예: 엘리베이터 버튼 여러 번 눌러도 한 번 부른 것과 같음)
- **왜 필요**: Kafka는 보통 **At-Least-Once**(최소 한 번) 전달이라 같은 메시지가 두 번 올 수 있다. 그대로 처리하면 알림이 중복 발송된다.
- **면접 단골**: "중복 메시지를 어떻게 막나?" → 고유 ID로 **이미 처리했는지 체크**(멱등 처리).
- **SafeAlert 연결**: Processor가 `alertId`로 중복 여부를 Redis(TTL)로 확인해 한 번만 처리.

## 6-5. 데이터 일관성: 강한 일관성 vs 최종 일관성

- **강한 일관성**: 쓰면 즉시 모두가 같은 값을 봄(정확하지만 느림/비용↑).
- **최종 일관성(Eventual Consistency)**: 잠깐은 다를 수 있지만 **시간이 지나면 일치**(MSA에서 흔히 선택).
- **SafeAlert 연결**: Outbox→Kafka 발행 사이 짧은 시간은 불일치하지만 결국 일치 → 최종 일관성.

## 6-6. Transactional Outbox 패턴 (★ 대표 설명거리)

- **문제**: "DB 저장"과 "Kafka 발행"을 한 트랜잭션으로 못 묶음 → DB는 됐는데 발행 실패 시 이벤트 유실.
- **해결**: 이벤트를 **같은 DB 트랜잭션 안의 Outbox 테이블**에 함께 저장(PENDING) → 별도 스케줄러가 폴링해 Kafka 발행 후 PUBLISHED로 변경.
- **효과**: Kafka가 죽어도 PENDING으로 남아 복구 후 재발행 → **유실 없음**(최종 일관성).
- **트레이드오프**: At-Least-Once라 소비자 **멱등 처리 필요**, Outbox 테이블 정리 필요.
- **SafeAlert 연결**: 구독 변경 이벤트에 적용. 장애 테스트에서 Kafka 중단→재시작 후 PENDING→PUBLISHED 전환 검증.

## 6-7. Circuit Breaker (회로 차단기)

- **개념**: 외부 의존(API·서비스)이 계속 실패하면 **회로를 열어(OPEN)** 호출을 즉시 차단하고 fallback을 반환 → 연쇄 장애 방지.
- **상태**: CLOSED(정상) → 실패율 초과 시 OPEN(차단) → 일정 시간 후 HALF-OPEN(재시도) → 성공 시 CLOSED.
- **면접 단골**: "외부 API 장애가 전체로 번지는 걸 어떻게 막나?" → 타임아웃 + Circuit Breaker + fallback.
- **SafeAlert 연결**: Collector가 공공 API 3종 호출. 한 곳이 죽어도 차단·스킵하고 나머지 2개는 정상 수집(단일 장애점 제거). **숨은 버그 경험**: API가 오류를 HTTP 200 + `resultCode`로 줘서 CB가 5xx만 보고 못 잡음 → 응답코드 검사 후 예외를 던지도록 수정.

## 6-8. Stateless 설계 (★ 트러블슈팅 핵심 교훈)

- **개념**: 서버가 **요청 간 상태(세션·캐시)를 자기 메모리에 저장하지 않음**. 상태는 외부(Redis/DB)에 둠.
- **왜 중요**: 서버를 여러 대로 늘리면(스케일아웃), 메모리에 상태가 있으면 인스턴스마다 달라 깨짐.
- **면접 단골**: "왜 stateless여야 확장이 쉬운가?" → 어느 서버가 요청을 받아도 동일하게 처리 가능 → 로드밸런싱·오토스케일링이 자유로움.
- **SafeAlert 연결(대표 스토리)**: Notification을 3대로 늘리니 알림 누락 발생 → WebSocket 세션이 서버 메모리에 있어 처리 서버≠연결 서버일 때 푸시 실패 → **Redis Pub/Sub**으로 모든 인스턴스에 브로드캐스트해 해결. "분산 환경에선 상태를 서버 메모리에 두면 안 된다"는 원칙을 체득.

## 6-9. 관측 가능성 (Observability)

분산 시스템은 어디서 문제가 났는지 알기 어렵다. 그래서 3가지를 본다.

- **메트릭(Metrics)**: 수치(요청 수, 응답 시간, 에러율, CPU) → **Prometheus + Grafana**.
- **로그(Logs)**: 텍스트 기록 → **ELK(Elasticsearch·Logstash·Kibana)**.
- **트레이싱(Tracing)**: 요청 하나가 여러 서비스를 거치는 **경로와 단계별 시간** → **Jaeger**(traceId를 서비스·Kafka 메시지까지 전파).
- **면접 단골**: "MSA에서 장애를 어떻게 추적하나?" → traceId로 분산 추적 + 중앙 로그 수집 + 메트릭 알람.

## 6-10. Kubernetes / 오토스케일링 (기초 용어)

- **컨테이너**: 앱 + 실행 환경을 한 박스로 포장(Docker). 어디서나 똑같이 실행.
- **Pod**: K8s에서 컨테이너가 실행되는 최소 단위.
- **Deployment / ReplicaSet**: 원하는 Pod 개수를 유지(죽으면 자동 재시작·복제).
- **Service**: Pod들에 대한 안정적인 진입점 + 로드밸런싱 + DNS.
- **HPA(Horizontal Pod Autoscaler)**: CPU 등 부하에 따라 **Pod 수를 자동 증감**.
- **면접 단골**: "K8s를 왜 쓰나?" → 자동 복구·확장·서비스 디스커버리·선언적 관리. Docker Compose로는 HPA·자동복구가 안 됨.
- **SafeAlert 연결**: minikube(AWS EC2)에 7개 서비스 배포, api-gateway HPA가 CPU 부하 시 **2→4로 자동 확장** 실측.
