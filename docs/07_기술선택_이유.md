# SafeAlert 기술 선택 이유

> 면접에서 자주 나오는 "왜 이 기술을 선택했나요?" 질문에 대한 답변 정리.
> 각 선택은 SafeAlert의 특성(재난 알림, 실시간, 다수 사용자)에 맞춰 판단했다.

---

## 1. MSA vs 모놀리식

### 선택: MSA (마이크로서비스 아키텍처)

**왜 MSA인가?**

SafeAlert는 기능별로 배포 주기와 스케일링 요구사항이 다르다.

| 서비스 | 특성 |
|--------|------|
| Alert Processor | 재난 발생 시 트래픽 급증 → Replica 3으로 확장 |
| Alert Collector | 1시간 주기 스케줄러 → 단독 배포 가능 |
| Auth Service | 인증은 변경 빈도 낮음 → 독립 운영 |
| Notification | WebSocket 연결 수 = 동시 접속자 수 → 별도 스케일링 |

모놀리식이라면 Alert Processor 하나를 스케일링하려 해도 전체 애플리케이션을 복제해야 한다.

**트레이드오프 (인정하는 단점)**

- 서비스 간 네트워크 통신 오버헤드 발생
- 분산 트랜잭션 처리 복잡 (Outbox 패턴으로 해결)
- 로컬 개발 시 여러 서비스를 동시에 실행해야 함

**결론**: 재난 알림처럼 트래픽이 예측 불가능하고 기능별 독립 배포가 필요한 서비스에는 MSA가 적합하다.

## 2. Kafka vs RabbitMQ

### 선택: Apache Kafka (KRaft 모드)

**왜 Kafka인가?**

| 비교 항목 | Kafka | RabbitMQ |
|---------|-------|---------|
| 메시지 보존 | 디스크에 영구 저장 (설정된 기간) | 소비 후 삭제 |
| 처리량 | 초당 수십만 건 | 초당 수만 건 |
| Consumer Group | 파티션 단위 병렬 처리 | Queue 경쟁 소비 |
| 재처리 | offset으로 과거 메시지 재소비 가능 | 소비 후 재처리 어려움 |

SafeAlert에서 결정적인 이유:

1. **Alert Processor Replica 3**: Kafka의 Consumer Group + 파티션 분산으로 3개 인스턴스가 메시지를 병렬 처리. RabbitMQ도 가능하지만 Kafka가 처리량에서 유리하다.

2. **메시지 유실 방지**: 재난 알림은 유실되면 안 된다. Kafka는 브로커가 일시 장애 후 복구돼도 메시지가 디스크에 남아 있어 재처리 가능하다.

3. **KRaft 모드**: Zookeeper 없이 단독 실행 가능 → 로컬 개발·K8s 배포 모두 단순화.

**트레이드오프**

- 설정과 운영이 RabbitMQ보다 복잡
- 소규모 서비스에서는 오버스펙일 수 있음

## 3. Transactional Outbox 패턴

### 선택 이유

구독 변경 이벤트를 Kafka로 직접 발행하면 아래 문제가 생긴다:

```
DB 저장 성공 → Kafka 발행 실패 → 이벤트 유실
```

Outbox 패턴은 DB 트랜잭션 안에 이벤트를 함께 저장한다:

```
DB 저장 + Outbox 테이블 저장 (같은 트랜잭션)
→ 스케줄러가 Outbox 폴링 → Kafka 발행 → PUBLISHED 상태로 변경
```

Kafka가 다운돼도 Outbox에 PENDING 상태로 남아 있다가, Kafka 복구 후 자동 재발행된다. 실제로 Phase 3 장애 테스트에서 Kafka 중단 → 재시작 후 PENDING → PUBLISHED 전환을 검증했다.

## 4. Redis 활용 전략

SafeAlert에서 Redis를 3가지 용도로 사용했다.

### 4-1. JWT Refresh Token 저장
```
key: refresh:{userId}
value: refreshToken
TTL: 7일
```
DB 조회 없이 O(1)으로 토큰 유효성 검증. 로그아웃 시 즉시 삭제.

### 4-2. Rate Limiting
```
key: ratelimit:{ip}
value: 요청 횟수 (INCR)
TTL: 1분
```
DB 조회 없이 분당 60건 초과 차단. 원자적 INCR로 동시성 문제 없음.

### 4-3. WebSocket 다중 인스턴스 동기화 (Pub/Sub)
Notification Service가 여러 인스턴스로 실행될 때, 한 인스턴스에 연결된 사용자에게 다른 인스턴스에서 메시지를 보낼 수 있어야 한다. Redis Pub/Sub로 인스턴스 간 메시지를 브로드캐스트한다.

## 5. WebSocket + STOMP

### 선택 이유

재난 알림은 서버→클라이언트 단방향 Push다. 선택지:

| 방식 | 특징 | 단점 |
|------|------|------|
| Polling | 구현 간단 | 지연 발생, 서버 부하 |
| SSE | 단방향 Push 특화 | 양방향 불가, 브라우저 연결 수 제한 |
| WebSocket | 양방향, 낮은 지연 | 구현 복잡 |

STOMP를 WebSocket 위에 올린 이유: 채널(/topic/public/alerts) 기반 구독 관리가 편리하고, Spring의 `@MessageMapping`으로 서버 구현이 깔끔해진다.

## 6. Circuit Breaker (Resilience4j)

### 선택 이유

Alert Collector가 기상청·행안부·환경부 3개 외부 API를 호출한다. 외부 API는 언제든 장애가 날 수 있다.

Circuit Breaker 없이 API 장애가 나면:
- 타임아웃까지 스레드 블로킹
- 연쇄 장애로 서비스 전체 영향

Circuit Breaker 적용 후:
- 실패율 50% 초과 시 즉시 차단 (OPEN)
- fallback 메서드로 빈 결과 반환 → 나머지 2개 API는 정상 수집 계속

Phase 3에서 실제로 API quota 초과 상황을 시뮬레이션해 Circuit Breaker 동작을 검증했다.

## 7. Kubernetes

### 선택 이유

| 기능 | 직접 구현 시 | K8s |
|------|------------|-----|
| Alert Processor 3개 실행 | 수동 관리 | `replicas: 3` 한 줄 |
| 서비스 장애 시 자동 재시작 | 별도 스크립트 | 자동 |
| 서비스 간 통신 | IP 하드코딩 | DNS(`service.namespace.svc.cluster.local`) |
| 모니터링 분리 | 설정 복잡 | `safealert-monitor` 네임스페이스 분리 |

포트폴리오 목적으로도, 실제 프로덕션 환경에 가장 가깝게 구성하는 방법이 K8s다.

## 8. PostgreSQL + MongoDB 혼용

### 왜 두 가지 DB를 쓰는가?

| 데이터 | DB | 이유 |
|--------|-----|------|
| 사용자, 구독, 알림 이력 | PostgreSQL | 관계형 데이터, 트랜잭션 필요 |
| 처리된 알림 원본 (event_store) | MongoDB | 스키마 유연, JSON 문서 그대로 저장 |

공공 API 응답은 기상청·행안부·환경부마다 구조가 달라 고정 스키마에 넣기 어렵다. MongoDB의 document 모델이 적합하다.

## 9. API Gateway (Spring Cloud Gateway)

### 선택 이유

인증 로직을 각 서비스에 분산하면:
- 코드 중복
- JWT 시크릿 키 관리 분산
- 인증 정책 변경 시 전 서비스 수정

Gateway에서 JWT 검증 후 `X-User-Id` 헤더로 전달하면 각 서비스는 헤더만 읽으면 된다. Rate Limiting도 Gateway 한 곳에서 처리한다.

Spring Cloud Gateway를 선택한 이유: Spring Boot와 동일한 생태계, WebFlux 기반으로 비동기 처리, 필터 체인 구성이 직관적이다.

---

## 10. JWT (Access + Refresh Token)

### 선택 이유

세션 방식 대신 JWT를 선택한 이유:
- MSA 환경에서 세션 공유는 중앙 세션 저장소 필요 (복잡도 증가)
- JWT는 자체 검증 가능 → Gateway에서 DB 조회 없이 검증

Access Token(15분) + Refresh Token(7일) 이중 구조:
- Access Token 짧게 → 탈취 시 피해 최소화
- Refresh Token Redis 저장 → 로그아웃 즉시 무효화 가능 (JWT 단점 보완)
