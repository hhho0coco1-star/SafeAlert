# SafeAlert 장애 테스트 결과

> Phase 3에서 수행한 장애 주입 테스트 결과 기록.
> 각 장애 상황에서 시스템이 어떻게 반응하는지, 복구 후 어떻게 정상화되는지 검증했다.

---

## 1. Kafka 브로커 중단

### 시나리오
- Subscription Service가 구독 이벤트를 Outbox 패턴으로 Kafka에 발행하는 중
- Kafka 브로커를 강제 중단

### 테스트 방법
```bash
# Kafka 컨테이너 강제 중단
docker stop safealert-kafka-1

# 구독 변경 API 호출 (이벤트 발행 시도)
curl -X POST http://localhost:8080/api/subscriptions/regions ...

# Kafka 재시작
docker start safealert-kafka-1
```

### 결과

| 단계 | 동작 |
|------|------|
| Kafka 중단 중 | 구독 저장은 성공 (DB 트랜잭션), Outbox에 PENDING 상태로 누적 |
| Kafka 재시작 후 | Outbox 스케줄러(5초 주기)가 PENDING 이벤트 감지 → Kafka 재발행 → PUBLISHED 전환 |
| 최종 결과 | **이벤트 유실 없음** |

### 핵심 코드 — Outbox 재시도 로직
```java
// OutboxScheduler.java
@Scheduled(fixedDelay = 5000)
public void publishPendingEvents() {
    List<OutboxEvent> pending = outboxRepository.findByStatus(PENDING);
    pending.forEach(event -> {
        try {
            kafkaTemplate.send(event.getTopic(), event.getPayload());
            event.setStatus(PUBLISHED);
        } catch (Exception e) {
            event.incrementRetryCount(); // 최대 3회 재시도
            if (event.getRetryCount() >= 3) event.setStatus(DEAD);
        }
        outboxRepository.save(event);
    });
}
```

### 결론
Transactional Outbox 패턴이 Kafka 장애 시 이벤트 유실을 완벽히 방지함을 검증.

## 2. Redis 장애 (Rate Limiting 비활성화)

### 시나리오
- API Gateway의 Rate Limiting이 Redis에 의존
- Redis 중단 시 Rate Limiting 동작 확인

### 테스트 방법
```bash
docker stop safealert-redis-1
curl http://localhost:8080/api/auth/login  # 요청 반복
docker start safealert-redis-1
```

### 결과

| 상황 | 동작 |
|------|------|
| Redis 중단 중 | Rate Limiting **비활성화** → 요청 정상 통과 (Fail-Open 방식) |
| 서비스 영향 | API Gateway는 정상 작동, 단 Rate Limiting 보호 해제 |
| Redis 복구 후 | Rate Limiting 자동 재활성화 |

### 핵심 코드 — Redis 장애 시 Fail-Open 처리
```java
// RateLimitFilter.java
.onErrorResume(e -> {
    log.warn("[RateLimit] Redis 연결 실패 — Rate Limiting 비활성화: {}", e.getMessage());
    return chain.filter(exchange); // 에러 시 요청 허용
});
```

### 결론
Redis 장애 시 Rate Limiting이 Fail-Open으로 처리되어 서비스 가용성을 우선함.
보안보다 가용성을 선택한 트레이드오프. 프로덕션에서는 Fail-Closed 전략도 고려 가능.

## 3. 공공 API 장애 (Circuit Breaker)

### 시나리오
- Alert Collector가 기상청·행안부·환경부 API 중 하나에 장애 발생
- API 응답 지연 또는 오류 시 Circuit Breaker 동작 확인

### 발견된 버그 및 수정

**초기 문제**: Circuit Breaker가 API 오류 시 작동하지 않음
- 원인: API가 HTTP 200으로 오류 코드를 반환 (예: `{"resultCode": "03"}`)
- Circuit Breaker는 HTTP 5xx만 감지 → 오류로 인식하지 못함

**수정 내용**: API 응답 코드 검사 후 명시적으로 예외 발생
```java
// DustAlertClient.java 수정 전
String response = restTemplate.getForObject(url, String.class);
// → 200 OK지만 실제론 오류인 경우 Circuit Breaker 미작동

// 수정 후
if (!"00".equals(resultCode)) {
    throw new RuntimeException("API 오류: " + resultCode); // CB가 감지 가능
}
```

### Circuit Breaker 동작 결과

| 단계 | 동작 |
|------|------|
| 정상 | API 호출 성공, CLOSED 상태 |
| 실패율 50% 초과 | Circuit Breaker OPEN → 즉시 fallback 반환 |
| fallback | `List.of()` 반환 → 해당 API 수집 스킵, 나머지 2개 API는 계속 수집 |
| 30초 후 | HALF-OPEN 상태로 전환, 재시도 |
| 재시도 성공 | CLOSED 상태 복귀 |

### 결론
3개 공공 API 중 하나가 장애여도 나머지 2개는 정상 수집. 단일 장애점 제거 확인.

## 4. 장애 테스트 종합 요약

| 장애 유형 | 영향 범위 | 복구 방식 | 데이터 유실 |
|---------|---------|---------|-----------|
| Kafka 브로커 중단 | 이벤트 발행 지연 | Outbox 스케줄러 자동 재발행 | 없음 ✅ |
| Redis 중단 | Rate Limiting 해제 | Redis 복구 시 자동 재활성화 | 없음 ✅ |
| 공공 API 오류 | 해당 API 수집 중단 | Circuit Breaker → fallback | 없음 ✅ |
