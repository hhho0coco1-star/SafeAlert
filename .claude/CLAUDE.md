# SafeAlert 프로젝트 규칙

## 작업 진행 방식 (필수 준수)

모든 작업(파일 생성, 코드 수정, 명령어 실행 등)은 반드시 아래 순서로 진행한다:

1. **착수 전 설명**: 다음에 무엇을 할지 먼저 설명한다
- 설명은 구체적으로 설명한다
2. **사용자 승인 대기**: 승인을 받은 후에만 실행한다
3. **완료 보고**: 작업 완료 후 결과를 요약해서 보고한다
- 완료 후 단계별로 어떠한 작업을 수행했는지 흐름 순으로 설명한다
4. **다음 단계 안내 후 대기**: 다음 단계를 설명하고 다시 승인을 기다린다

여러 단계를 연속으로 실행하지 말고, 반드시 각 단계마다 사용자의 확인을 받아야 한다.

## API 설계서 준수 규칙 (필수)

새로운 API를 구현하거나 기존 API를 수정할 때는 반드시 아래 절차를 따른다:

1. **설계서 먼저 확인**: `docs/03_API_DB설계.md`를 읽어 해당 API의 Request / Response 스펙을 확인한다
2. **DTO 일치 여부 검토**: 설계서의 `data` 필드와 실제 DTO 클래스의 필드가 일치하는지 대조한다
3. **불일치 발견 시 보고**: 설계서와 구현이 다르면 사용자에게 먼저 보고하고 어떻게 할지 결정을 받는다
4. **구현은 설계서 기준**: 특별한 이유가 없으면 구현을 설계서에 맞게 수정한다 (설계서가 기준)

### 현재까지 확인된 설계서 기반 응답 스펙

| API | data 필드 |
|-----|----------|
| POST /api/auth/signup | `{ userId, email, nickname }` |
| POST /api/auth/login | `{ accessToken, refreshToken, expiresIn: 900 }` |
| POST /api/auth/refresh | `{ accessToken, expiresIn: 900 }` |
| POST /api/auth/logout | `200 OK` (data 없음) |

## 튜토리얼 모드 (필수 준수)

사용자는 초보 개발자이므로 코드를 안내할 때 반드시 아래 방식을 따른다:

1. **코드를 한 덩어리씩**: 한 번에 5~10줄 단위로 나눠서 제공한다. 전체 코드를 한 번에 주지 않는다.
2. **왜 쓰는지 설명**: 각 코드 덩어리마다 이 코드가 왜 필요한지 쉬운 말로 설명한다. 전문 용어는 풀어서 설명한다.
3. **완료 대기**: 코드를 주고 나서 "입력하셨으면 '완료' 알려주세요"라고 하고 기다린다.
4. **완료 확인 후 진행**: 사용자가 "완료"라고 하면 다음 덩어리로 넘어간다.

## WBS 단계 완료 즉시 커밋 (필수 준수)

WBS 항목 하나가 완료될 때마다 다음 순서를 즉시 수행한다:

1. `docs/04_개발계획_WBS.md`에서 해당 항목을 `[ ]` → `[O]` 로 변경
2. `git -C "c:\study\SafeAlert" add .` 실행
3. `git -C "c:\study\SafeAlert" commit -m "[Phase X-X-X] 작업 내용"` 실행
4. `git -C "c:\study\SafeAlert" push origin main` 실행

다음 단계로 넘어가기 전에 반드시 위 4단계를 완료한다.

## /lego 커밋 자동 완료 (필수 준수)

`/lego` 명령어 실행 시 사용자에게 중간 확인을 구하지 않고 아래 순서를 자동으로 끝까지 완료한다:

1. `git status`로 변경 파일 확인
2. `git diff --stat`으로 변경 내용 파악
3. 한국어 커밋 메시지 작성 (형식: `[Phase X-X-X] 작업 내용`)
4. `git add .` 실행
5. 커밋 실행
6. `git push origin main` 실행
7. 결과 보고

중간에 "진행할까요?" 등의 확인 없이 1~7을 연속으로 실행한다.

## 챕터 완료 시 전체 검증 (필수 준수)

Phase(1-A, 1-B, 1-C 등) 하나가 끝나면 다음을 순서대로 수행한다:

1. **설계서 대조**: `docs/03_API_DB설계.md`를 읽고 해당 Phase에서 구현한 API들의 Request/Response 스펙 확인
   - Controller, Service, DTO 클래스가 설계서 스펙과 일치하는지 대조
   - 불일치 항목 발견 시 사용자에게 보고하고 수정 여부 결정

2. **컴파일 확인**: `./gradlew compileJava` 실행 → 오류 없으면 통과

3. **단위 테스트 실행** (테스트 파일이 있는 경우): `./gradlew test` 실행 → 결과 보고

4. **검증 완료 커밋**: 이상 없으면 `[Phase X 검증] 설계서 대조 및 테스트 완료` 형식으로 커밋 + push
