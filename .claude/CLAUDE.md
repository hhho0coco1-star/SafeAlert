# SafeAlert 프로젝트 규칙

## 1. 객관적 분석 (절대 규칙)
사용자 제안이든 AI 판단이든, 코드 작성·수정 전 항상 독립적으로 분석한다.
- 현업 기준으로 검토 → SafeAlert 상황(솔로, 포트폴리오)에 대입
- 더 나은 방법이 있으면 근거와 함께 제안, 현재 방법이 맞으면 "변경 불필요"라고 명확히 말한다
- 긍정 포장("좋아요!")으로 시작하지 말고 핵심 판단부터 말한다
- 한 번 내린 판단을 근거 없이 번복하지 않는다 (번복 시 이유 명시)
- 사용자가 못 본 예외·성능·보안 문제는 질문 범위를 넘어서라도 먼저 경고한다

## 2. 작업 진행 방식 (필수)
모든 작업은 [설명 → 승인 대기 → 실행 → 완료 보고 → 다음 단계 안내] 순서로 한 단계씩 진행한다.
여러 단계를 연속 실행하지 않고, 매 단계 승인을 받는다.

## 3. 튜토리얼 모드 (필수)
초보 개발자 대상이므로 코드는 5~10줄씩 나눠 제공하고, 각 덩어리가 왜 필요한지 쉬운 말로 설명한 뒤 "완료" 응답을 기다린다.

## 4. 파일 오타 체크 (필수)
사용자가 "ㅇㄹ"/"완료"로 입력 완료를 알리면, 다음 안내 전 직전 파일을 Read로 확인한다.
- 이상 없으면 "이상 없습니다." / 문제 있으면 줄 번호와 함께 지적

## 5. API 구현 규칙 (필수)
API 구현·수정 시 `docs/03_API_DB설계.md`를 먼저 읽고, DTO 필드가 설계서와 일치하는지 대조한다.
불일치 시 먼저 보고하고 결정을 받는다. (기준은 설계서)

## 6. WBS / todo.md 동기화 (필수)
작업 항목 변경 시 두 파일을 모두 갱신한다.

- `docs/04_개발계획_WBS.md` (영구 기록)
  - ①상단 전체 일정 개요 표 + ②해당 Phase 세부 테이블 **두 곳 모두**
  - 완료 항목은 `[ ]` → `[O]`로 표시 (제거하지 않음)

- `todo.md` (작업 큐)
  - 완료 항목은 **제거**
  - 새 작업은 우선순위 순으로, 파일명·구현 방향까지 구체화하여 추가

## 7. 커밋 규칙 (필수)
WBS 항목 하나가 완료되면 즉시 아래를 수행한다:
1. WBS 해당 항목 `[ ]` → `[O]`
2. `git -C "c:\study\SafeAlert" add .`
3. `commit -m "[Phase X-X-X] 작업 내용"` (한국어)
4. `push origin main`

**`/lego` 명령**: 위 절차를 중간 확인 없이 자동으로 끝까지 실행하고 결과만 보고한다.

## 8. Phase 완료 시 검증 (필수)
Phase 하나가 끝나면 순서대로 수행한다:
1. `docs/03_API_DB설계.md` 대조 (Controller/Service/DTO ↔ 설계서, 불일치 시 보고)
2. `./gradlew compileJava` (오류 없으면 통과)
3. 테스트 파일 있으면 `./gradlew test` 후 결과 보고
4. 이상 없으면 `[Phase X 검증] 설계서 대조 및 테스트 완료` 커밋 + push

> 개발 환경·DB 접속·포트·API 스펙은 `docs/05_개발환경.md` 참조
