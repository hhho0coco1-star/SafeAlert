[2026-05-22] WebSocket 다중 인스턴스 환경에서 알림 미수신 문제
────────────────────────────────────────────────────────────────────────────────

▶ 문제 상황
  - Notification Service 를 K8s 에 replica 3 으로 배포함
  - 공공 API → Kafka → Processor → Notification 파이프라인은 정상 동작
  - 그런데 브라우저에서 WebSocket 으로 연결해도 알림이 오지 않는 경우가 생김
  - 재현 조건: 알림이 처리된 서버 인스턴스와 사용자가 연결된 인스턴스가 다를 때

▶ 원인
  WebSocket 세션은 서버 메모리에 저장된다.
  즉 인스턴스 A 에 연결된 사용자 세션은 인스턴스 B 가 모른다.

  [그림]
  사용자 → 인스턴스 A 에 WebSocket 연결 (세션 저장)
  알림 처리 → 인스턴스 B 가 Kafka 메시지 소비
  인스턴스 B → 자신의 세션 목록에 해당 사용자 없음 → Push 못 함 ❌

  서버가 1대일 때는 이 문제가 없어서 초기에 발견하지 못했다.
  replica 를 3 으로 늘리고 나서야 간헐적으로 재현됐다.

▶ 해결 방법 — Redis Pub/Sub 도입
  어느 인스턴스가 알림을 처리하든 Redis 에 발행(Publish) 하면,
  모든 인스턴스가 Redis 를 구독(Subscribe) 하고 있다가
  자신에게 연결된 사용자에게 Push 한다.

  [그림]
  인스턴스 B 가 알림 처리
       ↓
  Redis "alert:public" 채널에 발행
       ↓
  인스턴스 A, B, C 모두 해당 채널 구독 중
       ↓
  인스턴스 A → 자기 세션에 연결된 사용자에게 WebSocket Push ✅

  Redis 를 라디오 방송국처럼 쓰는 방식이다.
  어느 서버에서 알림이 발생해도 Redis 에 한 번만 쏘면
  나머지 서버들이 자동으로 받아서 각자 연결된 사용자에게 전달한다.

  핵심 코드 흐름:
  1. AlertProcessedConsumer.java   — Kafka 메시지 소비 후 Redis "alert:public" 채널에 발행
  2. AlertPublicSubscriber.java    — Redis 채널 구독, 메시지 수신 시 SimpMessagingTemplate 으로 WebSocket 전송
  3. RedisConfig.java              — "alert:public" ChannelTopic 등록 + MessageListenerAdapter 연결

▶ 완료 기준
  [O] Redis Pub/Sub 채널 설계 (alert:public)
  [O] AlertPublicSubscriber.java 신규 생성
  [O] RedisConfig.java ChannelTopic 리스너 등록
  [O] 검증: 전국 알림 1건 수신 + 3개 인스턴스 모두 WebSocket 전달 확인

▶ 배운 점
  - 서버가 1대일 때 통과한 테스트가 replica 를 늘리면 깨질 수 있다
    → 분산 환경에서는 "상태(세션, 캐시)를 서버 메모리에 저장하면 안 된다"는 원칙이 있다
  - 이 원칙을 "무상태(Stateless)" 설계라고 부른다
    → 상태는 외부 저장소(Redis, DB)에 두고 서버는 요청만 처리
  - Redis Pub/Sub 은 메시지를 저장하지 않는다
    → 구독자가 연결되어 있을 때만 전달됨 (오프라인 사용자는 못 받음)
    → 알림 이력은 별도로 DB 에 저장해야 하는 이유

[2026-05-26] Jaeger 분산 트레이싱 — gRPC 수집 안 됨 → HTTP/protobuf 전환
────────────────────────────────────────────────────────────────────────────────

▶ 문제 상황
  - docker-compose 에 Jaeger all-in-one 컨테이너 추가 (포트 4317 gRPC)
  - 6개 서비스에 OpenTelemetry javaagent 설정 후 실행
  - Jaeger UI (localhost:16686) 에서 서비스 목록이 비어 있음
  - 트레이스 데이터가 전혀 수집되지 않음

▶ 원인
  Jaeger 에 데이터를 보내는 방법이 두 가지 있다.

  | 방법          | 포트 | 프로토콜    |
  |---------------|------|-------------|
  | gRPC          | 4317 | HTTP/2      |
  | HTTP/protobuf | 4318 | HTTP/1.1    |

  처음에 gRPC (4317) 로 설정했는데 데이터가 들어오지 않았다.

  원인은 두 가지가 겹쳤다:
  1. docker-compose 로컬 환경에서 HTTP/2(gRPC) 가 제대로 동작하지 않는 경우가 있음
  2. OpenTelemetry javaagent 기본값이 gRPC 인데,
     Jaeger 컨테이너 설정에서 gRPC 수신이 명시적으로 활성화되어 있지 않았음

▶ 해결 방법 — HTTP/protobuf (4318) 로 전환
  application.yml 의 otel 설정에서 엔드포인트 포트를 4317 → 4318 로 변경

  변경 전:
    otel.exporter.otlp.endpoint: http://localhost:4317

  변경 후:
    otel.exporter.otlp.endpoint: http://localhost:4318

  HTTP/protobuf 는 일반 HTTP/1.1 로 동작해서 로컬 환경 제약을 받지 않는다.
  변경 후 Jaeger UI 에서 5개 서비스 트레이스 수집 확인.

  [Jaeger 가 보여주는 것]
  요청 1건이 여러 서비스를 거칠 때 각 단계 소요 시간을 한눈에 볼 수 있다.

  예시:
  API Gateway (5ms) → Auth Service (28ms) → Notification Service (12ms)
                                              전체 응답: 45ms

  어느 서비스에서 병목이 생기는지, 에러가 어디서 났는지 추적할 수 있다.

▶ 완료 기준
  [O] docker-compose.yml Jaeger 컨테이너 추가 (16686, 4317, 4318 포트)
  [O] 6개 서비스 otel 엔드포인트 4317 → 4318 전환
  [O] Jaeger UI 에서 5개 서비스 트레이스 수집 확인

▶ 배운 점
  - gRPC 와 HTTP 는 같은 데이터를 다른 방식으로 전송한다
    gRPC = HTTP/2 기반, 빠르지만 환경 설정이 까다로움
    HTTP  = HTTP/1.1 기반, 느리지만 어디서든 잘 동작함
  - 로컬 개발 환경에서는 HTTP 방식이 더 안정적이다
    운영 환경(K8s) 전환 시 gRPC 로 바꾸는 것을 검토할 수 있음
  - "설정했는데 안 된다" 상황에서 디버깅 순서:
    1. 컨테이너 로그 확인 (docker logs jaeger)
    2. 포트 열려 있는지 확인
    3. 공식 문서에서 기본값 확인 → 내 설정과 비교
    4. 더 단순한 방법(HTTP) 으로 먼저 동작 확인 후 고급 방법(gRPC) 시도
